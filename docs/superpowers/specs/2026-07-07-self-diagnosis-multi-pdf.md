# Spec: 셀프진단(/d) 증권 여러 장 업로드 + 보험별 카드 보기 (PM 지시 2026-07-07)

> PM: "증권 PDF 여러 개 등록 가능하게, 하나씩 다 분석해서 카드 형식으로 보험이 먼저 나오고, 카드를 누르면 분석된 보험을 각각 볼 수 있게. 한 번에 하나씩만 보게."

## Decisions (locked)

### BE — `insurances/self_diagnosis.py` SelfDiagnosisView
1. **다중 파일 수신**: `request.FILES.getlist('files')` 지원 + 기존 단일 `file` 하위호환(있으면 files 목록에 합침). **1회 최대 5장**(초과분은 받지 않고 응답에 안내 문구 — 긍정 프레임 "증권은 한 번에 5장까지 정리해 드려요"). 파일 0장 = 기존 리드만 접수 플로우 불변.
2. **비용·남용 가드(기존 레일 재사용)**: 기존 per-ref 일일 캡·parse-attempt 카운터를 **파일 1장당 1회씩** 소모하도록 확장(요청당 1회가 아니라). 캡을 넘는 파일은 파싱하지 않고 해당 파일 항목에 "오늘은 여기까지 정리했어요. 담당 설계사가 나머지를 도와드릴 거예요" 상태로 표시(전체 요청은 실패시키지 않음).
3. **응답 형태(신규)**: 최상위 `insurances: [...]` 배열 — 항목별 `{name(상품명/파일명 폴백), company_label, monthly_premium, total_premium, coverage_count, tree(해당 보험만의 트리), status: 'ok'|'failed'|'skipped'}`. 파싱 실패 파일은 status='failed' + 긍정 안내("이 증권은 읽지 못했어요. 담당 설계사가 직접 확인해 드릴 거예요"). 기존 최상위 `tree`/`summary`(전체 합산)는 유지(하위호환·요약 카드용). 고객(리드)은 1명 생성, 성공한 보험 전부 그 고객에 귀속(기존 로직).
4. 동의 게이트 불변: PDF가 1장이라도 있으면 국외이전 동의 필수(기존 412 로직 그대로). 파싱은 파일별 독립 실패 격리(하나 죽어도 나머지 진행).

### FE — `app/d/[ref]/page.tsx`
1. **파일 선택 UI**: `<input multiple accept="application/pdf">` + 선택된 파일 목록(파일명 + 개별 제거 버튼) + "최대 5장" 안내. 기존 단일 선택 UX를 목록형으로 확장.
2. **분석 로딩**: 기존 신호등 애니메이션 유지, 문구에 장수 반영("증권 N장을 정리하고 있어요").
3. **결과 화면**:
   - 보험 1개 → 기존 상세 화면 그대로(변경 최소).
   - 2개 이상 → **보험 카드 목록 먼저**(카드: 보험명·회사·월보험료·담보 N개, 실패 카드는 안내 문구) → 카드 탭 → **그 보험의 보장 상세만**(보유 담보 리스트) + '목록으로' 뒤로가기. 한 번에 한 보험만 표시.
   - 상단 요약(월 보험료 합계·총 납입)은 성공한 보험 합산으로 유지. 예약/연락 CTA·면책 문구 기존 그대로.
4. 카피: 고객 대면 레드라인(§6) — 혜택+다음 행동, 판정어 0, em-dash·'준비 중' 금지(권유 단어 CI가 /d를 검사하므로 lint:copy 통과 필수).

## Tests (BE)
1. 2장 업로드 → insurances 2건, 각자 트리 분리, 고객 1명 생성, 동의 로그 정상.
2. 6장 업로드 → 5장만 처리 + 안내.
3. 1장 파싱 실패 + 1장 성공 → status 혼합, 요청은 201/200 유지.
4. 파일별 캡 소모: 남은 캡 1일 때 3장 → 1장 ok + 2장 skipped.
5. 기존 단일 `file` 필드 하위호환. PDF 없음 = 리드만(기존 테스트 불변).
6. 전체 스위트(550+) 그린.
