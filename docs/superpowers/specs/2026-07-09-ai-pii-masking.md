# AI 전송 전 신원정보 마스킹 (PM 확정: 가명처리 우선) — Spec

> 2026-07-09. PM 지시: AI(Claude) 사용 시 가장 중요한 건 가명처리·마스킹. 분석에 필요한 건 보장담보·금액·항목이지 이름·주민번호가 아니다. → **Claude 전송 전 신원식별정보를 마스킹/제거**하고, 담보 데이터만 보낸다. 우리 쪽은 어느 고객 문서인지 이미 알므로 재매핑 불필요.

## 핵심 사실 (확인됨)
- `claude_parser.py::claude_parse`가 pdfplumber 추출 **전체 텍스트(계약자 이름 + 주민번호 일부 포함)**를 그대로 Claude로 전송(`full_text` → user_prompt, :619-623). ocrparsing.py:927 주석에 "계약자\n박진희 (690913 - 2******)" 패턴 실재 → 이름·주민번호가 실제로 국외 전송됨.
- 분석(담보 인식)은 이름/주민번호 불필요 — 담보명·금액·기간만 필요. OCR은 특정 고객 대상으로 시작되므로 **AI가 신원을 되돌려줄 필요 없음**(우리가 이미 앎).

## 설계 (개인정보 최소화 = PIPA 국외이전 최소화, 컴플라이언스 POSITIVE)

### 1. `_strip_identity(text) -> text` (core/ocr, Claude 전송 직전)
Claude로 보낼 텍스트에서 신원식별자만 **보수적·정밀 패턴**으로 마스킹(담보 데이터는 절대 미변경):
- **주민등록번호**: `\d{6}\s*[-–]\s*[1-4][\d*]{6}`(완전/부분 마스킹형) → `******-*******`.
- **전화번호**: `01[016789][-\s.]?\d{3,4}[-\s.]?\d{4}` → `010-****-****`.
- **이메일**: 표준 이메일 패턴 → `***@***`.
- **증권/계약 번호**: 필요시(식별 위험 낮으면 보류). 이번엔 주민·전화·이메일 우선.
- **계약자/피보험자 성명**: ★ 가장 조심 — 한글 이름은 담보명·일반어와 구분 어려움. **라벨 근접만 마스킹**: `(계약자|피보험자|수익자|성명|피보험자명)\s*[:\n]\s*[가-힣]{2,4}` 형태에서 이름 토큰만 → `***`. ocrparsing.py의 기존 이름 추출 로직(NOT_NAMES 등 :935)을 재사용해 라벨 근접 이름만 식별. **담보명(암진단비 등)·회사명·금액은 절대 미마스킹.**
- 반환은 마스킹된 텍스트. 원본은 우리 DB(고객 레코드)에 그대로(마스킹은 Claude 전송분에만).

### 2. 적용 지점 (모든 Claude 호출)
- `claude_parse`(OCR): `full_text = _strip_identity('\n'.join(text_lines))` 후 전송.
- `verify.py`(교차검증): 동일.
- `self_diagnosis`(공개 /d): 동일(리드 신원은 우리 폼에서 별도 수집, Claude엔 마스킹본).
- `compare _generate_guide_draft`: rows(담보명+금액)만 보내므로 신원 PII 없음 → 확인만(마스킹 불필요).

### 3. 재매핑 불필요 (coverage 분석)
- 담보 분석은 특정 고객 대상 → AI 응답의 담보 결과를 **이미 아는 고객**에 그대로 귀속. 토큰화-복호화 왕복 불필요(단순 STRIP). PM 원안('되돌릴 때 우리가 매핑')을 더 단순·안전하게: 애초에 신원을 안 보냄.

### ★ 리스크 & 검증 (money-path)
- **최대 리스크: 과다 마스킹으로 담보 데이터 손실** → 담보 인식 정확도 하락. 방어: (1) 보수적 정밀 패턴(주민번호·전화·이메일은 형태가 명확, 오탐 거의 없음), (2) 이름은 라벨 근접만, (3) **기존 OCR 파싱 테스트 + mock-SDK 테스트로 담보 인식 불변 실증**(마스킹 전후 담보 결과 동일).
- 마스킹 함수 유닛 테스트: 주민번호/전화/이메일/라벨근접이름 마스킹 + 담보명(암진단비)·금액(1000만원)·회사명 **미변경** 확인.
- PII 로그 레드라인(§7) 유지: 마스킹 실패해도 로그에 원문 안 남김.

### 컴플라이언스 노트 (PM 정직 브리핑)
- 이름/주민번호를 안 보내는 것 = 국외 전송 개인정보 최소화 = 강한 PIPA 자세. **좋은 방향.**
- ★ 단, **병력(건강정보)을 AI에 보내는 경우**엔 이름을 가려도 병력 내용 자체가 민감정보 → 국외이전 동의는 여전히 필요(CPO 판단). 이번 마스킹은 '일반 담보 분석'의 신원 최소화이고, 병력 분석 기능(ANALYZE_MEDICAL_ENABLED, 닫힘)은 별개로 동의+CPO 검토 유지.
- 고객 대면 무변경. 마이그레이션 0(전송 텍스트 가공만).
