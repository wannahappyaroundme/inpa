# 여러 증권 시각 비교 문구 전환 설계

## 목표

인파의 비교 기능을 `현재 보험과 새 제안 비교`가 아니라 `여러 증권의 담보·보장금액·보험료를 같은 기준으로 시각화하는 기능`으로 일관되게 설명한다. 사용자가 보험 교체 권유나 특정 계약 선택을 인파가 주도한다고 받아들일 수 있는 표현은 사용자 화면에서 제거한다.

## 확정된 방향

- 사용자 화면의 비교 기준은 항상 `증권 A`와 `증권 B`다.
- `현재/제안`, `갈아타기/승환`, `비교안내서` 표현은 사용자 화면에서 제거한다.
- 기능명은 문맥에 따라 `증권 비교`, `여러 증권 비교`, `증권 비교표`로 통일한다.
- 비교 결과는 담보, 보장금액, 보험료와 그 차이를 표와 그래프로 보여주는 데 집중한다.
- 어느 증권이 더 낫다는 판단, 보험 교체를 전제로 한 유의사항, 고객 발행을 전제로 한 표현은 비교 화면에 노출하지 않는다.
- DB 필드, API 응답, 기존 `portfolio_type`, `ai_compare` 같은 내부 식별자는 호환성을 위해 유지한다.

## 검토한 접근

### 1. 발견된 문구만 부분 교체

변경량은 가장 작지만 랜딩, 온보딩, 실제 비교 화면, 요금제, FAQ에 기존 표현이 다시 남을 가능성이 높다. 서비스 설명이 화면마다 달라질 수 있어 채택하지 않는다.

### 2. 사용자 노출면 전체 전환, 내부 구조 유지

공개 랜딩과 실제 서비스 화면을 같은 원칙으로 바꾸고, 데이터 구조는 그대로 둔다. 중개로 오해될 표현을 제거하면서 회귀 위험을 낮출 수 있어 이 방식을 채택한다.

### 3. 데이터 모델과 API까지 전면 개명

`보유/제안` 구분과 비교 API의 내부 이름까지 바꾸면 의미는 가장 깔끔하지만 마이그레이션과 대규모 회귀 위험이 생긴다. 사용자 가치에는 영향이 없어 이번 범위에서 제외한다.

## 사용자 노출 문구 기준

| 기존 표현 | 변경 원칙 |
| --- | --- |
| 비교 분석 | 증권 비교 또는 여러 증권 비교 |
| 현재와 제안을 나란히 비교 | 여러 증권을 같은 기준으로 나란히 확인 |
| 현재 / 제안 | 증권 A / 증권 B |
| 제안서 업로드 | 증권 추가 |
| 보유(기존 가입) / 제안(갈아타기) | 비교 묶음 A / 비교 묶음 B |
| 비교안내서 | 증권 비교표 |
| 유지·전환 변동 비교 | 담보·보험료 차이 비교 |
| 해지 손실·면책기간 리셋 중심 예시 | 증권별 보험료·담보 차이 예시 |

`보험 분석`, `보장 분석`, 실제 문서 종류로서 백엔드가 인식하는 `가입제안서`는 비교 권유와 다른 개념이므로 내부 파싱 코드에서 유지한다.

## 적용 범위

### 공개 화면

- 기본 랜딩의 히어로, 기능 카드, 예시 카드, 사용 방법, 차별점, 요금제, 하단 설명
- 영화형 랜딩의 기능 설명, 실제 화면 캡션과 대체 텍스트
- 검색 설명, 앱 메타데이터, 설치형 앱 설명
- FAQ와 온보딩의 비교 기능 설명
- 관리자 데모에서 설계사에게 보여주는 기능명과 설명

### 로그인 후 서비스

- 고객 상세 탭 이름을 `증권 비교`로 변경한다.
- 비교 열 이름은 데이터 종류와 관계없이 항상 `증권 A`, `증권 B`로 표시한다.
- 보험 선택 안내는 `A/B 비교 묶음`으로 설명한다.
- 비교 화면에서 올리는 문서는 `증권 추가`로 표현한다.
- 직접 입력 화면의 사용자 선택지는 `비교 묶음 A/B`로 표시한다.
- 복사 기능은 `증권 비교표 내용 복사`로 표현하고 담보·보험료 사실만 포함한다.
- 보험 교체를 전제로 계산한 경고와 `AI 비교안내서`, `발행` 영역은 렌더링하지 않는다. 관련 백엔드 코드와 게이트는 제거하지 않는다.
- 요금 한도와 업그레이드 안내는 `증권 비교` 기준으로 표시한다.

### 운영 데이터와 문서

- 신규 가입자가 보는 공지·FAQ 시드의 비교 설명을 같은 기준으로 바꾼다.
- `README.md`는 PM이 이해할 수 있도록 현재 기능 설명을 갱신한다.
- `AGENTS.md`는 사용자 노출 정책과 내부 호환성 결정을 개발 정본에 반영한다.
- 과거 의사결정 기록과 법무 검토 문서는 역사적 근거이므로 일괄 치환하지 않는다.

## 변경하지 않는 것

- 증권 자동 인식, 개인정보 마스킹, 담보 정규화와 금액 계산
- 여러 보험을 A/B 묶음에 자유롭게 넣는 현재 비교 방식
- DB 스키마, 저장된 보험의 `portfolio_type`, API 필드명
- 내부 법무 주석과 비활성 기능의 백엔드 게이트
- 비교 결과를 어느 쪽이 낫다고 판단하지 않는 기존 원칙

## 오류·빈 상태

- 보험이 없으면 `증권을 등록하면 비교할 수 있어요`라고 다음 행동을 안내한다.
- 한쪽 묶음이 비어 있으면 A와 B에 각각 한 개 이상의 증권을 고르도록 안내한다.
- 로딩, 다시 불러오기, 한도 안내도 `증권 비교` 용어로 통일한다.
- 기존 요청 순서 보호와 오류 처리 동작은 변경하지 않는다.

## 검증 기준

1. 공개 랜딩, 온보딩, FAQ, 고객 상세, 직접 입력, 업그레이드 화면에서 `현재/제안` 대립 표현과 `갈아타기`, `승환`, `비교안내서`가 노출되지 않는다.
2. 고객 상세 비교표는 모든 조합에서 `증권 A/증권 B`를 사용한다.
3. 비교 결과는 담보·보장금액·보험료의 표와 그래프만 보여준다.
4. 복사되는 내용에는 판단, 권유, 보험 교체 전제 문구가 없다.
5. 프런트 문구 검사, 전체 단위 테스트, 프로덕션 빌드가 통과한다.
6. 백엔드 설정 검사와 비교 관련 테스트가 통과한다.
7. 데스크톱과 모바일에서 랜딩과 고객 상세 비교 화면을 실제로 확인한다.

## 배포 원칙

- 구현은 최신 `master`에서 분기한 `codex/multi-policy-copy`에서 진행한다.
- 코드 검증과 리뷰를 마친 뒤 PR을 만든다.
- 운영 배포는 별도 확인을 받고 진행한다.
