# 보험증권 추출·검토·분석 정확도 아키텍처 — Design

> 2026-07-16. PM 승인 방향: 보험 업로드·표시·분석을 인파의 최우선 핵심 기능으로 둔다. 실제 설계사 피드백이 없는 현재 단계에서는 Claude 한 공급자만 사용한다. 로컬 코드 검증으로 의심 항목을 찾고, 설계사가 원문과 비교해 수정·확정한 데이터만 분석·공유한다. GPT 추가와 이중 공급자 비교는 실제 오류 자료와 설계사 피드백이 쌓인 뒤 필요성을 다시 판단한다. 목표는 AI 자동 인식률 100%를 주장하는 것이 아니라, 확인되지 않은 중대한 오류와 조용한 담보 누락이 고객 공유로 넘어가는 비율을 0으로 만드는 것이다.

## 1. 결정 요약

1. **지원 입력은 우선 전자 PDF다.** 스캔 PDF와 휴대폰 사진은 이번 범위에 넣지 않는다.
2. **현재 Claude 파이프라인을 유지한다.** 이번 구현에는 OpenAI API와 두 번째 공급자를 추가하지 않는다.
3. **검증은 코드 규칙과 사람 확인이 담당한다.** 날짜·금액·중복·누락 후보·근거 존재 여부를 서버가 검사하고, 의심 항목을 설계사에게 분명히 알린다.
4. **AI의 신뢰도 자기평가는 사용하지 않는다.** 원문 근거와 코드 규칙으로 `확인할 항목`을 만들며, 규칙을 모두 통과해도 사람이 최종 확인하기 전에는 확정값으로 취급하지 않는다.
5. **AI 결과는 초안이다.** 업로드가 성공해도 보험·담보 최종 레코드로 즉시 확정하지 않는다.
6. **설계사가 검토 완료한 보험만 분석에 포함한다.** 분석은 기존 규칙·합산 엔진이 담당하며 LLM이 금액을 계산하거나 보장 등급을 정하지 않는다.
7. **공유는 확정된 불변 자료만 허용한다.** 링크 발급과 스냅샷 생성을 한 트랜잭션으로 묶고, 고객 링크는 현재 DB를 다시 계산하지 않는다.
8. **GPT와 이중 공급자는 후속 검토 항목이다.** 실제 설계사 수정 로그와 반복 오류가 쌓여 단일 공급자의 한계가 확인될 때만 비교 실험을 별도 설계한다.
9. **동시 요청은 설계사·고객·작업 단위로 완전히 격리한다.** 여러 설계사가 같은 시각에 파일을 올려도 파일, 상태, Claude 결과, 검토 초안, 분석, 공유본이 다른 계정에 연결될 수 없어야 한다.

## 2. 목표와 비목표

### 2.1 목표

- 보험사, 상품명, 월 보험료, 계약일, 만기일, 담보 원문명, 가입금액, 갱신 여부를 원문 근거와 함께 추출한다.
- 원문에 있는 담보가 파싱 과정에서 사라지지 않게 한다.
- 동일 파일 재시도와 기존 증권 교체를 구분해 중복 합산을 막는다.
- Claude 초안·원문 근거·코드 검증 결과를 설계사가 한 화면에서 확인하고 수정한다.
- 검토 완료된 보험만 기존 한눈표·보험료 요약·공유 화면의 입력으로 사용한다.
- 모델·프롬프트·결과 형식·정규화 사전 버전을 기록해 회귀와 향후 모델 변경을 측정한다.
- 실패와 비용을 개인정보 없이 운영 화면에서 확인한다.
- 여러 설계사의 동시 업로드와 같은 고객의 여러 증권 처리를 안전하게 직렬화·병렬화한다.

### 2.2 비목표

- 사진 촬영, 이미지 PDF, 필기 인식
- 의료정보 분석과 자동 상품 추천
- LLM이 적정 보장금액이나 가입·교체 권고를 판단하는 기능
- 보험사 전산과의 계약 원장 대조
- 원본 PDF의 장기 보관
- GPT·다른 AI 공급자 추가와 공급자 간 자동 비교

## 3. 현재 구조의 문제

현재 흐름은 다음과 같다.

```text
전자 PDF
→ pdfplumber 텍스트 추출
→ 신원정보 마스킹
→ Claude 구조화
→ CustomerInsurance·CustomerInsuranceDetail 즉시 생성
→ 선택적 Claude 재검증
→ 분석·공유
```

문제는 모델 종류보다 결과 확정 방식에 있다.

- 업로드 성공과 데이터 검토 완료가 구분되지 않는다.
- 서버가 `verification`과 `created_cases`를 반환해도 화면이 사용하지 않는다.
- 금액이 없거나 매칭되지 않은 담보가 고객별 검토 목록에 보존되지 않을 수 있다.
- 동일 PDF 재업로드를 막는 파일 지문이 없다.
- 보험·담보를 수정하고 재분석하는 검토 화면이 없다.
- 담보 매핑 수정이 전역 `InsuranceDetail.analysis_detail`에 영향을 줄 수 있다.
- 분석과 공유가 검토 완료·해지·분석 제외 상태를 엄격히 필터링하지 않는다.
- 공유 스냅샷은 존재하지만 공개 링크가 현재 DB를 다시 계산하고, 스냅샷 생성 실패도 링크 발급을 막지 않는다.
- Claude의 `confidence`나 현재 선택적 재검증 결과를 화면이 최종 확정 근거로 사용할 수 없다.
- 동기 요청 안에서 최대 120초짜리 AI 호출을 처리해 웹 워커 고갈과 타임아웃 위험이 있다.

## 4. 검토한 접근

| 접근 | 장점 | 한계 | 결정 |
|---|---|---|---|
| Claude 단독 + 즉시 저장 | 변경이 가장 작음 | 현재처럼 누락·오류가 최종 데이터로 바로 들어갈 수 있음 | 폐기 |
| Claude 단독 + 코드 검증 + 사람 확정 | 현재 자산을 유지하면서 실제 오류와 수정 피드백을 수집할 수 있음 | 설계사 확인 단계가 필요하고 자동 처리율이 낮아질 수 있음 | **권장·승인** |
| GPT 또는 이중 공급자 | 공급자별 오류 비교 가능성 | 지금은 실제 설계사 오류 자료가 없어 복잡도·비용·동의를 먼저 늘리게 됨 | 실제 피드백 이후 재검토 |

구조를 지킨 JSON도 값 자체는 틀릴 수 있다. 따라서 현재 단계의 권위는 모델이 말하는 확신이 아니라 원문 근거, 코드 규칙, 설계사의 최종 확인이다.

## 5. 목표 데이터 흐름

```text
파일 선택
→ 형식·크기·동의·중복 사전 확인
→ 비공개 임시 저장 + 작업 ID 발급(202)
→ 워커가 페이지별 텍스트와 줄 ID 추출
→ 신원정보 마스킹
→ Claude 구조화
→ 결과 정규화
→ 코드 검증·누락 후보·원문 근거 검사
→ 검토 초안 저장
→ 의심 항목 표시
→ 설계사가 모든 값을 원문과 비교해 수정·확정
→ 기존 보험·담보 모델로 한 번에 반영
→ 확정 보험만 코드 기반 분석
→ 불변 공유 스냅샷 생성
→ 고객 링크는 해당 스냅샷만 조회
```

### 5.1 파일 접수

- 지원 형식은 암호화되지 않은 전자 PDF다.
- 파일 헤더와 MIME을 함께 검사하며 확장자만 믿지 않는다.
- 서버가 스트리밍으로 SHA-256을 계산한다.
- 같은 고객·같은 파일 지문의 진행 중 또는 검토 중 작업이 있으면 새 보험을 만들지 않고 기존 작업을 반환한다.
- 기존 확정 보험과 같은 파일이면 `기존 자료 보기`를 안내한다. 새 버전은 설계사가 `기존 자료 교체` 또는 `새 보험 추가`를 고른다.
- 업로드 API는 작업 ID와 상태를 `202 Accepted`로 즉시 반환한다.

### 5.2 임시 원본과 작업 큐

현재 Claude 호출도 긴 동기 웹 요청에 넣지 않는다.

- 비공개 R2 임시 경로에 원본을 저장한다. 공개 URL은 만들지 않는다.
- 별도 작업 워커가 파일을 읽고 처리한다. 권장 구성은 Celery + Redis/Valkey + Render background worker다.
- 원본은 검토 확정·취소 시 즉시 삭제한다. 미완료 작업은 `INSURANCE_SOURCE_RETENTION_HOURS` 후 자동 삭제하며 기본값은 24시간이다.
- 삭제 작업은 시스템이 찍은 `source_expires_at`과 `confirmed_at/canceled_at` 조건을 함께 확인한다.
- 원본 장기 보관은 이번 범위가 아니다. 24시간이 지난 검토를 이어갈 때는 같은 파일을 다시 선택하고 SHA-256으로 기존 초안에 연결한다.

### 5.3 페이지·줄 근거

로컬 추출 결과를 다음 형태로 정규화한다.

```json
{
  "line_id": "p03-l014",
  "page": 3,
  "line": 14,
  "text_masked": "일반암진단비 30,000,000원"
}
```

- 원본 텍스트는 메모리에서 신원정보 로컬 보완에만 사용한다.
- AI와 영속 초안에는 마스킹된 줄만 전달·저장한다.
- 모든 추출 필드는 `evidence_line_ids`를 필수로 반환한다.
- 근거 줄이 없거나 근거와 값이 맞지 않으면 자동 일치로 처리하지 않는다.

### 5.4 Claude 단일 추출

- Claude는 마스킹 원문, 엄격한 JSON Schema, 담보 분류 설명을 받는다.
- 모델 ID는 `CLAUDE_MODEL_PARSE` 환경변수로만 주입한다. 코드 기본 모델 ID를 두지 않는다.
- 추출기는 `extract(masked_lines, schema_version) -> ExtractionResult` 계약을 갖는다.
- 프롬프트는 문서 텍스트를 신뢰할 수 없는 데이터로 취급하고 문서 안의 명령을 무시한다.
- Claude SDK 재시도는 네트워크·타임아웃·429·5xx에만 최대 3회, 1초→2초→4초로 적용한다. 400·401·403은 재시도하지 않는다.
- 기존 선택적 Claude 재검증 결과는 추가 경고 자료로만 사용할 수 있다. 같은 공급자의 두 번째 답변이므로 정확성 확정 근거가 될 수 없다.

### 5.5 항목별 확인 상태

날짜·공백·쉼표·금액 단위·보험사 별칭을 코드로 정규화한 뒤 원문 근거와 규칙을 검사한다.

| 상태 | 조건 | 처리 |
|---|---|---|
| `review_ready` | 원문 근거가 있고 모든 코드 규칙 통과 | 사람이 원문과 맞는지 확인 후 확정 |
| `needs_review` | 값·담보 수·표준 위치가 불명확 | 확인할 항목 최상단 |
| `no_evidence` | 값은 있으나 원문 근거 없음 | 확인 필요 |
| `unmatched` | 원문 담보는 있으나 표준 위치 미확정 | 삭제하지 않고 직접 배정 필요 |
| `invalid` | 날짜·금액·합계 규칙 위반 | 수정 전 확정 금지 |
| `manual` | 설계사가 직접 입력·수정 | 다른 항목과 함께 최종 확인 필요 |

모델이 반환하는 `confidence` 값은 의사결정에 사용하지 않는다. 화면의 상태는 근거·규칙·사람 확인으로만 만든다. `review_ready`도 `정확함`을 뜻하지 않으며, 설계사의 최종 확인 전에는 분석과 공유에 들어가지 않는다.

- `needs_review | no_evidence | unmatched | invalid` 항목은 행 단위로 수정·배정·제외 사유를 확정해야 한다.
- `review_ready` 항목은 행마다 누르게 만들지 않고, 설계사가 전체 표를 검토한 뒤 `증권 원문과 같은지 확인했습니다`를 체크해 한 번에 확인할 수 있다.
- 최종 확인 체크와 버튼은 같은 화면에 두며, 체크 전에는 확정 API를 호출할 수 없다.
- 이 확인은 책임 전가 문구가 아니라 AI 초안을 실제 보험 데이터로 바꾸는 업무 단계다.

### 5.6 코드 검증

LLM 출력 후 별도 검증기가 다음을 검사한다.

- 날짜 형식과 계약일≤만기일 관계
- 금액이 음수가 아니고 원문 근거에 같은 숫자 또는 명확한 단위 변환이 있는지
- 같은 원문 담보의 중복 생성 여부
- 담보 후보 행 수와 추출·미매칭·제외 행 수가 보존되는지
- 보험별 월 보험료와 담보별 보험료 합계가 비교 가능한 문서에서 일치하는지
- 갱신·비갱신 구분과 납입기간 조합이 유효한지
- 보험사 코드와 상품 종류가 모순되지 않는지
- `created + unmatched + intentionally_excluded = detected_candidates` 보존식

검증기는 값을 추측해 고치지 않는다. 불일치를 검토 항목으로 만든다.

날짜와 납입기간은 다음 단일 권위를 추가로 따른다.

- 고객 생년월일, 계약일, 만기일은 `YYYY-MM-DD`를 정식 형식으로 사용하고,
  기존 자료의 `YYYY.MM.DD`도 같은 공용 파서로 읽는다. 존재하지 않는 날짜는 형식이
  비슷해도 확정 전에 차단한다.
- 담보 보험료가 있는 갱신형은 손해보험에서 유효한 계약일·만기일과 올바른 순서가
  있어야 한다. 생명보험 계산이 고객 나이를 사용하면 유효한 생년월일도 필요하다.
- 기간 정보가 부족해 총 갱신 개월을 계산할 수 없으면 `0원`을 만들지 않고 미상으로
  남기며 확정 전에 사람이 날짜를 확인하게 한다.
- 납입기간 `종신`은 `년`, `세`, `년 갱신`과 다른 권위값이다. 담보의
  `payment_period_type=4`로 보존하고 숫자 납입기간을 함께 저장하지 않는다.
- 종신납의 총 납입보험료는 계산식이 확정되지 않은 값이므로 `None`을 유지한다.
  월 보험료는 알고 있는 사실로서 갱신·비갱신 월 합계에 계속 포함할 수 있다.
### 5.7 다중 설계사 동시 처리 격리

모든 처리 단위의 정본 키는 `owner_id + customer_id + job_uuid`다. 파일명, 화면에서 선택한 고객 이름, 워커 실행 순서, 전역 최근 작업 값으로 결과를 연결하지 않는다.

- 임시 파일 키는 `insurance-imports/<owner_id>/<customer_id>/<job_uuid>/source.pdf`처럼 작업 UUID까지 포함한다.
- Claude 호출의 입력·메타·응답 객체는 작업 함수 안의 지역 변수로만 유지한다. 모듈 전역 dict, 최근 결과 캐시, 공유 임시파일 경로를 두지 않는다.
- 워커는 작업을 가져올 때 `job_uuid + attempt_uuid + lease_expires_at`을 기록한다. 늦게 끝난 이전 시도가 새 시도의 결과를 덮어쓸 수 없다.
- 큐 메시지에는 `job_uuid`만 넣는다. owner, customer, 파일 경로는 워커가 DB에서 다시 읽고 검증하며, 요청 본문이나 큐 메시지의 경로 값을 신뢰하지 않는다.
- 워커는 DB의 owner/customer/job과 임시 저장 경로의 네임스페이스가 일치하지 않으면 처리하지 않고 보안 오류로 종료한다.
- 결과 저장은 `WHERE id=<job_uuid> AND attempt_uuid=<현재 시도> AND status=<기대 상태>` 조건의 compare-and-set으로 처리한다.
- 작업·결과·검토 API는 URL의 job ID만 믿지 않고 항상 `owner=request.user`와 `customer__owner=request.user`를 함께 검사한다.
- 같은 파일 지문은 **같은 설계사·같은 고객·같은 보유/제안 구분 안에서만** 중복으로 본다. 다른 설계사가 동일한 보험사 양식을 올린 경우는 서로 독립된 작업이다.
- 한 고객에게 서로 다른 증권 여러 개를 동시에 올리는 것은 허용한다. 각 파일은 별도 작업으로 처리하고 확정 시 고객 행을 잠근 뒤 최신 분석 버전 위에 순서대로 반영한다.
- 같은 기존 보험을 두 작업이 동시에 교체하려 하면 먼저 확정한 작업만 성공한다. 뒤 작업은 `409 IMPORT_TARGET_CHANGED`로 최신 자료를 다시 확인하게 한다.
- 확정 API는 `select_for_update`로 job, 대상 customer, 교체 대상 insurance를 잠그고 보험·담보·분석 버전 반영을 한 트랜잭션으로 수행한다.
- 임시 원본 삭제는 해당 job의 정확한 storage key만 삭제한다. 디렉터리 접두사 일괄 삭제로 다른 작업 파일을 지우지 않는다.
- 분석 캐시를 사용한다면 키에 `owner_id`, `customer_id`, `analysis_version`을 모두 포함한다. owner 없는 고객 ID 단독 캐시는 금지한다.
- 진행 상태는 DB의 해당 job 행이 정본이다. 웹 프로세스 메모리나 브라우저 타이머를 정본으로 사용하지 않는다.
- 한 설계사가 작업 큐를 독점하지 않도록 사용자별 동시 실행 상한과 전체 상한을 분리한다. 초기값은 실측 부하테스트로 정하며 관리자 설정으로 조절한다.

동시 처리의 원칙은 “병렬로 읽고, 고객 데이터 반영은 잠금과 버전으로 안전하게 순서화”다.

## 6. 데이터 모델

### 6.1 `InsuranceExtractionJob` 신규

한 번의 파일 접수·처리를 나타내는 owner 전용 레코드다.

- `id` UUID
- `owner` FK(User, CASCADE)
- `customer` FK(Customer, CASCADE)
- `portfolio_type`
- `status`: `queued | extracting | validating | review_required | confirmed | failed | canceled | superseded`
- `file_sha256`, `file_size`, `page_count`, `safe_display_name`
- `source_storage_key` nullable, 절대 API 응답에 포함하지 않음
- `source_expires_at`, `source_deleted_at`
- `masked_lines` JSON 또는 압축 텍스트
- `draft_payload` JSON
- `validation_summary` JSON
- `schema_version`, `prompt_version`, `normalization_version`
- `created_at`, `started_at`, `completed_at`, `confirmed_at`, `canceled_at`
- `error_code` enum, `error_type` enum
- `attempt_uuid`, `lease_expires_at`, `draft_version`, `target_insurance_version` nullable

제약:

- `(owner, customer, file_sha256, portfolio_type)`의 활성 작업 중복을 조건부 유니크 제약으로 막는다. 활성 상태는 `queued | extracting | validating | review_required`로 고정한다. owner가 다르면 같은 파일 지문이어도 충돌하지 않는다.
- `attempt_uuid`와 `lease_expires_at`은 워커가 작업을 claim할 때 함께 채우고, 재시도 claim마다 새 attempt UUID로 바꾼다.
- 원문·프롬프트·AI 원문 응답은 로그에 남기지 않는다.
- `draft_payload`는 마스킹된 보험 사실만 포함한다.

### 6.2 `InsuranceExtractionResult` 신규

Claude 추출 결과와 운영 지표를 작업 상태에서 분리한다.

- `job` FK
- `provider`: `claude | local_regex`
- `model_id`, `outcome`, `structured_payload`
- `input_tokens`, `output_tokens`, `estimated_cost_krw`, `latency_ms`
- `created_at`

`structured_payload`는 owner 전용이며 원문 전체나 식별정보를 포함하지 않는다. `(job, provider)`는 1건으로 수렴한다. 향후 다른 공급자를 검토하더라도 이번 구현에서는 OpenAI 결과를 생성하지 않는다.

### 6.3 `CustomerInsurance` 확장

- `review_status`: `legacy_review_required | confirmed | excluded | superseded`
- `source_job` nullable FK
- `confirmed_at`, `confirmed_by`
- `analysis_included` Boolean, 기본 False
- `confirmation_source`: `planner_review | manual_entry | legacy_review`

기존 보험은 정확성 주장 없이 `legacy_review_required`, `analysis_included=False`로 데이터 이관한다. 기능 게이트를 켜기 전 설계사가 기존 자료를 한 번 확인할 수 있는 전환 화면을 제공한다. 신규 업로드는 작업 초안에서 확정할 때만 `CustomerInsurance`를 생성하거나 명시적으로 교체한다.

### 6.4 `CustomerInsuranceDetail`의 개별 매핑 근거

- `raw_name` 유지
- `source_page`, `source_line_start`, `source_line_end`, `source_text_masked`
- `review_reason`
- `mapping_source`: `global | planner_override | manual`
- `analysis_detail_override` M2M
- `confirmed_at`
- `payment_period_type`: `1=년 | 2=세 | 3=년 갱신 | 4=종신`

분석 엔진은 `mapping_source=planner_override`일 때 해당 케이스의 override를 사용한다. 설계사의 한 고객 수정이 전역 `InsuranceDetail.analysis_detail`을 바꾸지 않는다. 전역 사전 개선은 별도 관리자 검수 큐로만 반영한다.

### 6.5 미매칭 보존

전역 `UnmatchedLog`만으로는 특정 고객 검토에 부족하다. `draft_payload.coverages[]`에 모든 원문 행을 남기고, 확정 전까지 각 행을 다음 중 하나로 끝낸다.

- 표준 담보에 배정
- 분석 제외(사유 필수)
- 중복으로 제외(대상 행 필수)
- 문서 머리글·설명문으로 제외(근거 필수)

이 상태가 정해지지 않은 행이 하나라도 있으면 확정할 수 없다.

## 7. API 계약

- `POST /api/v1/customers/<id>/insurance-imports/`
  - 파일·추가/교체 의도·idempotency key
  - `202 {job_id, status}`

  - 처리 상태, 진행 단계, 안전한 오류 코드

  - 보험 기본정보와 담보 행 수정·추가·배정·제외
  - 낙관적 잠금용 `draft_version` 필수
- `POST /api/v1/insurance-imports/<job_id>/confirm/`
  - 미해결 항목 0건·규칙 통과·`planner_confirmed_source_match=true` 확인
  - `draft_version`과 교체 대상 보험 버전 필수
  - 한 트랜잭션으로 보험·담보 생성/교체·계산·작업 확정·임시 원본 삭제 예약
- `POST /api/v1/insurance-imports/<job_id>/cancel/`
  - 작업 취소와 임시 원본 삭제
- 기존 수기 CRUD
  - 보험 기본정보뿐 아니라 담보 행을 추가·수정·삭제할 수 있게 확장
  - 마지막 단계에서 같은 확정 검증기를 사용

모든 API는 customer owner 스코프와 `IsEmailVerified`를 적용한다. 작업 ID가 타인의 것일 때는 404로 존재를 숨긴다. 동일 작업의 중복 PATCH·confirm은 idempotency key와 버전 검사로 한 결과에 수렴한다.

## 8. 화면 설계

### 8.1 정보 위계

고객 분석 탭의 행동 순서를 다음으로 고정한다.

1. `증권 등록` 또는 `확인 이어하기`
2. 처리 중 작업 카드
3. 보험별 검토 카드
4. 보험·담보 수정과 검토 완료
5. 보험료 요약과 담보 한눈표
6. 고객에게 보일 화면 미리보기
7. 공유와 공유 기록

예약·동의·공유 기록 같은 보조 행동은 주 행동과 같은 무게로 나열하지 않는다.

### 8.2 검토 화면

- 상단: 파일명, 보험사·상품명, 처리 시각, 읽은 담보 수, 확인할 항목 수
- 데스크톱: 왼쪽 원본 PDF, 오른쪽 검토 항목의 나란히 보기
- 모바일: 각 항목의 `원문 보기`를 누르면 해당 페이지를 여는 전체 화면 보기
- 기본정보: 보험료·기간·갱신 여부를 원문 근거와 함께 편집
- 안내: `자동으로 정리한 내용이에요. 증권 원문과 같은지 직접 확인해 주세요.`
- 담보 표: 원문 담보명, 가입금액, 표준 위치, 확인 상태, 근거 페이지
- 기본 정렬: `확인할 항목 먼저`
- 행 행동: 수정, 표준 위치 변경, 분석 제외, 삭제 취소
- 하단 고정 행동: `검토 완료하고 분석에 반영`
- 확인할 항목이 남으면 `확인이 필요한 항목 N개`와 첫 항목 이동을 제공
- 코드 규칙을 모두 통과한 경우에도 `정확함`이나 `검증 완료`라고 표시하지 않는다. 설계사가 최종 버튼을 누른 뒤에만 `확인 완료`로 바뀐다.
- 최종 체크: `증권 원문과 같은지 확인했습니다`

원본 보기는 owner 인증을 거친 짧은 만료 URL만 사용하고 브라우저 캐시를 막는다. 원본이 24시간 보존기간을 지나 삭제된 경우에는 같은 파일을 다시 선택하면 SHA-256으로 기존 초안에 연결해 검토를 이어간다.

사용자 문구에는 `OCR`, `JSON`, `모델 합의`, `컴플라이언스` 같은 기술·운영 용어를 쓰지 않는다.

### 8.3 상태와 오류

- 처리 중: 실제 작업 상태를 표시하고 가짜 퍼센트를 만들지 않는다.
- Claude 결과는 있으나 근거·규칙이 부족함: `확인할 항목이 있어요`로 안내하고 해당 값을 직접 수정할 수 있게 한다.
- Claude 호출 또는 구조화 실패: 보험 레코드를 만들지 않고 같은 파일 재시도·직접 입력을 제공한다.
- 담보 0건: 성공으로 표시하지 않는다.
- 동의 필요: 파일을 전송하기 전에 동의 링크를 제공하고, 동의 완료 후 같은 선택 흐름으로 이어간다.
- 중복 파일: 새 항목을 만들지 않고 기존 결과를 연다.
- 네트워크 실패와 데이터 없음은 다른 화면으로 구분한다.

모바일 320px, 200% 확대, 담보 100개, 긴 보험명, 키보드·VoiceOver/NVDA 흐름을 출시 검증에 포함한다. 처리·오류·완료 상태는 `aria-live` 또는 적절한 alert/status 역할로 전달한다.

## 9. 분석·공유 권위

### 9.1 분석 대상

다음 조건을 모두 만족한 보험만 합산한다.

- `portfolio_type=1`
- `review_status=confirmed`
- `analysis_included=True`
- `is_cancelled=False`
- `superseded`가 아님

미확정·해지·분석 제외 보험은 목록에는 사실 상태로 보이되 한눈표 합계에는 들어가지 않는다. 분석 화면은 포함 보험 수, 제외 보험 수, 마지막 확정 시각, 확인할 항목 수를 표시한다.

### 9.2 계산

- 보장금액·보험료 합계는 기존 Python 계산 엔진이 수행한다.
- LLM은 원문을 구조화하고 담보 위치 후보를 제안할 뿐 합계를 만들지 않는다.
- 기준선이 없으면 현재와 같이 보유 사실만 표시한다.
- 결과 금액을 누르면 보험→원문 담보→가입금액의 합산 근거를 펼친다.

### 9.3 공유

- 공유 생성 전에 모든 포함 보험이 확정 상태인지 서버가 검사한다.
- 토큰 회전, 스냅샷 생성, 이벤트 기록을 한 DB 트랜잭션으로 처리한다.
- 스냅샷 생성이 실패하면 링크도 발급하지 않는다.
- 공유 토큰은 하나의 `ShareSnapshot`을 가리킨다.
- 공개 `/s/<token>`은 저장된 `payload`만 반환하며 현재 고객 데이터를 재계산하지 않는다.
- 새 분석을 공유하면 새 스냅샷과 토큰을 발급하고 이전 링크는 회수 또는 만료 정책을 따른다.
- 동의 철회·고객 삭제·공유 회수 시 토큰과 스냅샷을 함께 닫는다.

이 결정은 `2026-07-08-share-snapshot-archive.md`의 “스냅샷 실패가 링크 발급을 막지 않음”을 정확도 우선 원칙에 따라 대체한다.

### 10.1 현재 동의 범위

향후 GPT 또는 다른 공급자를 검토할 때는 별도 설계로 다음을 먼저 완료해야 한다.

1. 새 공급자의 수신자 법인·처리 국가·전송 항목·목적·보유 정책 확인
2. 동의문과 개인정보처리방침의 수탁자 고지 변경
3. `CONSENT_TEXTS_VERSION` 상향과 기존 고객 재동의
4. 개인정보 제거 평가셋에서 기존 Claude 대비 오류 검출 이득 확인

### 10.2 데이터 최소화

- Claude는 `_strip_identity`를 통과한 텍스트만 받는다.
- 이름·주민번호·전화·이메일은 AI에 보내지 않는다.
- 프롬프트·응답 원문·PDF 내용은 로그와 추적 도구에 남기지 않는다.
- 운영 로그는 job/user ID, 모델 ID, 토큰, 비용 추정, 지연, outcome enum, 담보 수만 기록한다.
- API 키는 서버 비밀변수로만 관리하고 `NEXT_PUBLIC_*`에 두지 않는다.
- 임시 원본은 private 저장소, owner API, 짧은 보존기간, 다중 삭제 가드를 적용한다.
- Claude 장애가 기본 분석·기존 확정 데이터 조회를 깨뜨리지 않는다.

### 10.3 기능 게이트

- `INSURANCE_REVIEW_GATE_ENABLED=False`
- `INSURANCE_SOURCE_RETENTION_HOURS=24`

코드 기본값은 새 검토·공유 게이트를 열지 않는다. 스테이징 검증과 PM 승인 후 운영 환경에서 단계별로 연다.

## 11. 평가 설계

### 11.1 평가 자료

- 개인정보를 제거하고 정답을 사람이 확정한 실제 전자 PDF 최소 100건
- 담보 행 최소 1,000건
- 생명·손해보험, 주요 보험사, 오래된·새로운 양식, 짧은·긴 증권을 층화
- 평가셋은 git에 넣지 않고 접근 통제된 저장소에 보관
- 학습·프롬프트 개선용 세트와 최종 holdout 세트를 분리

### 11.2 비교 대상

- A: 현재 pdfplumber → Claude → 즉시 저장
- B: pdfplumber → Claude → 원문 근거·코드 검증 → 설계사 수정·확정

같은 Claude 모델, 결과 형식, 정규화 버전을 사용해 검토 단계가 실제 오류를 얼마나 막는지 측정한다. 실제 설계사 피드백이 쌓이기 전에는 GPT 비교를 평가 범위에 넣지 않는다.

### 11.3 지표

- 보험사·상품명 exact match
- 보험료·가입금액·날짜 exact match
- 담보 행 recall: 실제 담보 중 빠지지 않고 잡은 비율
- 담보 행 precision: 실제 없는 담보를 만들지 않은 비율
- 표준 담보 위치 정확도
- 조용한 누락 건수
- 오류·불확실 항목을 검토 화면으로 보낸 비율
- 검토 후 중대한 금액 오류 건수
- 처리 p50/p95, Claude 실패율, 건당 추정 비용
- 설계사 검토 시간과 수정 행 수

### 11.4 출시 차단 기준

다음 중 하나라도 발생하면 해당 단계는 운영에 열지 않는다.

- holdout에서 조용히 사라진 담보 1건 이상
- 미확정·해지·분석 제외·교체된 보험이 합산 또는 공유됨
- 같은 파일이 중복 합산됨
- Claude 결과를 사람 확인 없이 최종값으로 확정함
- 원문 근거 없는 금액이 분석에 포함됨
- 공유 링크 내용과 발급 스냅샷이 다름
- 스냅샷 실패 후 링크가 발급됨
- 현재 버전의 고객 본인 동의가 없는 자료가 Claude로 전송됨
- 원문·프롬프트·응답 또는 신원정보가 로그에 남음
- 한 설계사의 파일·진행 상태·초안·분석·공유본이 다른 설계사의 작업에 연결됨
- 늦게 끝난 워커가 재시도 또는 다른 작업의 최신 결과를 덮어씀

0건의 오류가 100% 정확성을 통계적으로 증명하지는 않는다. 따라서 배포 뒤에도 보험사·양식별 표본 검수와 회귀 평가를 계속한다.

## 12. 운영 계측

기존 Claude 비용 화면에 `추출 품질·검토` 지표를 확장한다.

- 모델·보험사·양식별 성공률
- 검토 준비·확인 필요·근거 없음·미매칭 비율
- 담보 후보 수와 최종 확정 수
- 설계사 수정률과 검토 완료 시간
- 토큰·추정 비용·p50/p95 지연
- 대기 작업 수, 사용자별 대기시간, 작업 재시도율, lease 만료율
- `입력 문서는 있는데 감지 담보 0건`과 같은 부재 알림
- 특정 보험사에서 누락률이 기준을 넘으면 해당 양식을 전체 수동 확인 대상으로 바꾸는 운영 스위치

원문과 고객 식별정보는 계측하지 않는다. 운영자가 결과 예시를 확인해야 할 때는 별도로 비식별화된 평가 케이스만 사용한다.

## 13. 단계별 출시

### Phase A. 비식별 기준선 평가

- 현재 Claude 기준선 고정
- 줄 근거·코드 검증이 포함된 새 결과 형식 작성
- 100건·1,000행 골든셋으로 현재 방식과 새 검토 방식 측정
- 고객 데이터 전송 없음, 제품 화면 변화 없음

### Phase B. 작업·검토 기반

- 비동기 작업, 중복 방지, 검토 초안, 미매칭 보존
- 보험·담보 수정, 케이스별 매핑 override
- 기존 보험의 `legacy_review_required` 전환 화면
- 검토 완료 보험만 분석하는 게이트는 스테이징에서 먼저 검증

### Phase C. 실제 설계사 피드백 학습

- 설계사 수정 전·후 값을 개인정보 없이 구조화해 오류 유형 집계
- 보험사·양식·담보명별 반복 오류를 사전·규칙·프롬프트에 반영
- 설계사가 `확인 필요` 표시를 이해하고 실제 원문 확인에 사용하는지 인터뷰·행동 데이터로 검증
- 단일 Claude에서 반복적으로 검출하지 못하는 오류가 확인될 때만 GPT·다른 공급자 비교안을 새로 설계

### Phase D. 불변 공유

- 확정 데이터만 공유
- 토큰이 ShareSnapshot을 직접 가리키도록 변경
- 공개 링크·공유 기록·회수·만료·철회 일치 검증

### Phase E. 모델·비용 재평가

정확도 기준을 충족한 뒤에만 검토한다.

- 실제 반복 오류가 있을 때 GPT 또는 다른 모델 비교 필요성
- 충분한 오류 자료가 쌓인 뒤 낮은 비용 모델 적용 가능성
- 프롬프트 캐시와 배치 처리

비용 최적화로 정확도 또는 오류 검출률이 내려가면 적용하지 않는다.

## 14. 검증 체크리스트

### Backend

- 중복 파일과 idempotency 동시 요청이 작업 1건으로 수렴
- 작업 워커 재시도·중단·재시작 후 상태 복구
- 서로 다른 설계사 20명이 같은 시각에 각 3개 PDF를 올려도 모든 결과가 원래 owner/customer/job에만 귀속
- 같은 파일 hash를 서로 다른 설계사가 올리면 독립 작업 2건, 같은 설계사·고객이 재시도하면 작업 1건
- 늦게 끝난 attempt가 최신 attempt 결과를 덮어쓰지 못함
- 같은 고객의 서로 다른 증권 동시 확정은 모두 보존되고, 같은 보험 동시 교체는 한 건만 성공한 뒤 다른 건이 409
- 작업·초안·원본 보기·확정 API에서 타 설계사 접근이 모두 404
- 한 작업의 취소·만료 정리가 다른 작업의 R2 원본을 삭제하지 않음
- Claude 구조화 결과와 코드 검증 결과의 분리
- Claude 성공·실패·빈 결과의 상태 전이
- 모든 담보 후보의 보존식 검증
- draft_version 충돌 409
- 확정 트랜잭션 롤백과 원본 삭제 예약
- owner 스코프 404
- 분석 쿼리가 확정·포함·정상 보험만 선택
- 공유 생성·스냅샷·토큰 원자성
- 공개 링크가 DB 수정 후에도 같은 payload 반환
- 동의 철회·회수·만료·고객 삭제 후 링크 종료
- 임시 원본 24시간 파기와 다중 삭제 가드
- PII 로그 회귀 테스트

### Frontend

- 업로드 전 형식·크기·동의·중복 안내
- 실제 처리 상태와 새로고침 후 복구
- 검토 준비·확인 필요·미매칭·오류 행 편집
- 확인할 항목이 남으면 확정·공유 차단
- 담보 추가·삭제·위치 변경 후 분석 재계산
- 빈 상태·부분 실패·전체 실패·재시도
- 고객이 실제로 볼 불변 화면 미리보기
- 320px, 200% 확대, 키보드, 스크린리더

### Runtime

- 스테이징에서 실제 API 호출과 작업 완료 확인
- 마이그레이션 적용 후 SELECT로 제약·인덱스 확인
- Render web/worker와 Redis 장애·재시작 시험
- 동시 60건 업로드 부하에서 교차 귀속 0건, 중복 생성 0건, 큐 공정성·p95 확인
- 토큰·비용·지연 계측 확인
- 실제 공개 링크 curl과 브라우저 렌더 확인
- 프로덕션 배포는 별도 PM 승인 후 수행

## 15. 완료 정의

이 기능은 코드가 작성됐을 때가 아니라 다음이 모두 증명됐을 때 완료다.

- 비식별 holdout 평가 결과와 실패 사례가 문서화됨
- 모든 원문 담보 후보가 확정·제외·중복 중 하나로 설명됨
- 확인 전 보험은 분석과 공유에 들어가지 않음
- 설계사가 보험·담보를 수정하고 근거를 확인할 수 있음
- 같은 파일이 중복 합산되지 않음
- 동시 요청에서 설계사·고객·작업 간 교차 귀속이 0건임
- 케이스별 수정이 다른 고객 데이터에 영향을 주지 않음
- 공유 링크가 발급 당시 화면과 영구히 일치함
- Claude 전송 전에 현재 버전 고객 본인 동의가 서버에서 검증됨
- 원문과 PII가 로그에 남지 않음
- 백엔드 테스트, 프론트 빌드, 런타임 API, 브라우저 검증이 통과함
