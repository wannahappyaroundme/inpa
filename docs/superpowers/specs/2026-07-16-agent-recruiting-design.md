# 설계사 영입과 Manager 자동 승격 설계

> 확정일: 2026-07-16<br>
> 상태: PM 대화 기준 제품 설계 확정, 구현 전 문서 검토 단계<br>
> 기능명: **설계사 영입**<br>
> 핵심 대상: 다른 GA 또는 원수사에서 활동 중인 경력 보험설계사를 영입하려는 설계사와 조직 리더

## 1. 한 문장 정의

인파의 `설계사 영입`은 설계사가 알고 지내는 경력 설계사에게 개인 영입 링크를 보내는 관계형 영입에서 시작해, SNS·채용 채널 캠페인과 인파의 공개 콘텐츠 유입까지 확장하고, 영입된 설계사의 초기 정착을 1·4·8·13주에 짧게 지원하는 별도 업무 흐름이다.

보험 가입자인 고객을 관리하는 기능과는 화면, 데이터, 검색, 통계, 권한, API를 완전히 분리한다.

## 2. 최종 제품 결정

### 2.1 첫 출시 방향

세 가지 접근의 평균 점수는 다음과 같다.

| 안 | 설명 | 사업·제품·리스크 그룹 평균 |
|---|---|---:|
| A | 영입 후보 관리 중심 | 86.7점 |
| **B** | **영입 후보 관리 + 최소 정착 지원** | **88.0점** |
| C | 인파가 직접 운영하는 커리어·매칭 플랫폼 | 51.7점 |

**B를 선택한다.** 후보 연락 관리만 제공하면 일반 채용 관리 도구와 차이가 작다. 반대로 인파가 여러 GA를 연결하는 플랫폼까지 한 번에 운영하면 현재 사업의 범위를 벗어난다. 후보 영입과 1·4·8·13주의 최소 정착 지원을 같은 흐름으로 연결하면 현장 문제를 더 깊게 해결하면서도 인파의 기존 팀 초대, 알림, Manager 현황 기능을 재사용할 수 있다.

### 2.2 기능의 위치

- `설계사 영입`은 고객 관리의 부속 메뉴가 아닌 1차 메뉴다.
- 메뉴명은 `팀 성장`이나 `후임 양성`이 아니라 **설계사 영입**으로 통일한다.
- 내부 탭은 다음 네 개다.
  1. `영입 현황`
  2. `나의 영입 페이지`
  3. `캠페인 링크`
  4. `정착 지원`
- 후보가 실제 팀원이 된 이후에는 기존 Manager 현황과 연결하되, 위촉 전 후보 기록은 고객이나 팀 성과 데이터에 합치지 않는다.

## 3. 대상 사용자와 해결할 일

### 3.1 Plus 설계사

- 알고 지내는 다른 GA·원수사 경력 설계사에게 자신의 영입 페이지를 보낸다.
- 후보가 관심을 보이면 연락할 사람과 다음 행동을 놓치지 않는다.
- 본인이 데려온 첫 설계사가 인파 팀에 합류하면 Manager로 자동 승격된다.
- 이후 본인도 SNS와 채용 채널별 링크를 만들어 영입 활동을 넓힌다.

### 3.2 Manager

- 자신과 팀원이 영입 중인 후보 흐름을 본다.
- 후보 상세정보는 담당 설계사 중심으로 제한하고, 조직 전체 화면은 집계를 기본으로 한다.
- 위촉 후 1·4·8·13주에 활동 상태와 필요한 지원을 빠르게 확인한다.
- 수수료 계보가 아니라 연락 누락, 상담 전환, 초기 정착을 관리한다.

### 3.3 경력 설계사 후보

- 소개한 리더와 조직의 실제 정보를 읽는다.
- 대화를 신청하고 연락받을 시간대를 고른다.
- 같은 연락처로 여러 링크를 열었을 때 최종적으로 대화할 리더를 직접 선택한다.
- 연락 중단과 정보 삭제를 요청할 수 있다.

## 4. 단계별 확장 순서

### 4.1 1단계: 관계형 영입

첫 사용 흐름은 이미 알고 지내는 경력 설계사에게 개인 링크를 보내는 방식이다.

- 설계사별 공개 영입 페이지 `/r/<token>`
- 카카오톡, 문자, 메신저 복사를 위한 짧은 소개 문구
- 후보의 대화 신청
- `새 신청 → 연락 → 대화·면담 → 위촉 준비 → 팀 합류` 단계 관리
- 다음 행동과 다음 확인일
- 실제 인파 팀 초대 수락 연결
- 팀 합류 후 1·4·8·13주 정착 확인

### 4.2 2단계: 설계사별 공개 캠페인

설계사가 Threads, TikTok, Instagram, 사람인, 잡코리아 등에서 직접 영입 활동을 할 수 있게 채널별 링크를 제공한다.

- 채널별 캠페인 링크와 QR
- 승인된 소개 문구 조합
- 방문, 신청, 연락, 면담, 팀 합류 전환 구분
- 같은 공개 영입 페이지를 재사용하되 캠페인 출처만 분리
- 채널별 성과는 본인과 권한 있는 Manager에게 집계로 제공

인파가 외부 사이트의 후보 정보를 수집하거나 자동으로 대량 연락하지 않는다. 설계사가 직접 게시한 콘텐츠와 후보의 자발적 신청만 받는다.

### 4.3 3단계: 인파 공개 콘텐츠 유입

인파가 경력 설계사에게 도움이 되는 공개 콘텐츠와 광고로 새로운 후보를 모은다.

- 인파의 경력·이직 관련 공개 콘텐츠
- 경력 설계사의 자발적 대화 신청
- 후보가 직접 리더를 비교하고 한 명을 선택
- 선택한 리더에게만 후보 상세정보 전달
- 유입 경로와 선택 결과를 인파 운영 화면에서 집계

인파가 후보를 임의로 여러 GA에 동시에 배포하지 않는다. 여러 리더가 같은 후보를 영입하려는 경우에도 최종 담당 리더는 후보의 명시적 선택으로 결정한다.

## 5. Plus와 Manager의 가격·승격 원칙

### 5.1 하나의 Plus, 두 가지 성장 상태

요금은 역할에 따라 오르지 않는다.

| 사용자 상태 | 화면 표시 | 월 요금 | 열리는 경험 |
|---|---|---:|---|
| 개인 활동 | `Plus for 설계사` | 19,900원 | 개인 영업과 설계사 영입 |
| 첫 팀원 합류 이후 | `Plus for Manager` | 19,900원 | 개인 기능 + 팀·정착·Manager 현황 |

- 부가세 표시는 기존 요금 정책을 그대로 따른다.
- 별도 Manager 추가 결제는 만들지 않는다.
- 첫 팀원이 합류해도 결제일, 구독 만료일, 연간 결제 기간, 쿠폰, 사용량은 초기화하지 않는다.
- 신규 구매 화면에서는 Plus와 Manager를 서로 다른 가격 상품처럼 판매하지 않는다.
- 가격 화면에서는 같은 Plus가 설계사의 성장에 맞춰 확장된다는 관계를 보여준다.

### 5.2 자동 승격의 정확한 기준

후보 카드에서 `팀 합류` 단계를 수동으로 선택한 것만으로는 Manager 기능을 열지 않는다. 첫 설계사가 인파 팀 초대를 수락해 활성 팀 관계가 만들어지는 순간 승격한다.

승격 조건은 다음과 같다.

1. 초대한 사용자와 합류한 사용자가 서로 다른 인증 계정이다.
2. 팀 초대 토큰이 유효하다.
3. 합류한 설계사의 `manager` 관계가 서버에 저장된다.
4. 초대한 사용자의 최초 Manager 승격 시각을 한 번만 기록한다.
5. 활성 Plus 구독자는 같은 구독을 유지한 채 Manager 기능 권한을 얻는다.

승격은 상태 변경 이벤트로 기록하고 중복 처리에도 한 번만 적용한다. 팀원이 나중에 0명이 되어도 승격 기록과 기존 팀 데이터는 유지한다. Plus 구독이 끝난 경우 데이터는 보존하고, 다시 Plus를 시작하면 같은 Manager 상태로 이어서 사용한다.

### 5.3 사용자에게 보이는 승격 경험

승격 직후 한 번만 축하 안내를 보여준다.

> **Manager로 승격되었어요**<br>
> 첫 팀원이 합류해 팀 관리 기능이 열렸습니다. 추가 결제 없이 계속 이용할 수 있어요.

안내의 다음 행동은 `팀 현황 보기`와 `다음 설계사 영입하기` 두 가지다. 승격은 과장된 계급 표현보다 새로운 책임과 도구가 열렸다는 느낌으로 전달한다.

### 5.4 기존 Manager 가입자 처리

- 기존 `Manager` 구독 기록은 삭제하거나 일괄 변경하지 않는다.
- 신규 결제 화면은 Plus 중심으로 통합한다.
- 기존 Manager 구독자는 동일한 금액, 만료일, 쿠폰, 기능을 유지한다.
- 서버는 기존 Manager 구독과 새 `Plus + Manager 자격`을 모두 같은 팀 기능 권한으로 해석한다.
- 운영자가 기존 기록을 확인할 수 있도록 관리자 화면에는 원래 플랜 코드와 현재 역할을 함께 보여준다.

## 6. 핵심 사용자 흐름

### 6.1 관계형 영입 링크 생성

1. 설계사가 `설계사 영입` 메뉴에 들어온다.
2. `나의 영입 페이지`에서 사진, 소속, 직책, 활동 지역, 지원 방식을 확인한다.
3. 자유로운 수익 홍보 문구 대신 승인된 문장 중에서 실제 지원 내용을 선택한다.
4. 개인 링크와 복사용 소개 문구를 만든다.
5. 알고 지내는 경력 설계사에게 직접 보낸다.

### 6.2 후보의 대화 신청

1. 후보가 공개 페이지에서 리더와 팀 정보를 확인한다.
2. 이름, 휴대전화, 경력 구분, 현재 또는 최근 소속(선택), 활동 지역, 연락 가능 시간대를 입력한다.
3. 개인정보 수집·이용과 영입 상담 연락에 동의한다.
4. `대화 신청하기`를 누른다.
5. 신청 완료 화면에서 연락 주체와 다음 진행을 확인한다.
6. 담당 설계사는 인앱 알림을 받고 `영입 현황`의 새 신청에서 후보를 확인한다.

### 6.3 후보 진행 관리

단계는 다음과 같이 고정한다.

| 단계 | 의미 | 필수 다음 행동 |
|---|---|---|
| 새 신청 | 후보가 신청을 마침 | 24시간 안에 첫 연락 |
| 연락 | 첫 연락을 시작함 | 대화·면담 일정 확인 |
| 대화·면담 | 영입 대화를 진행함 | 위촉 준비 여부 확인 |
| 위촉 준비 | 서류·소속 이동을 준비함 | 팀 초대 전달 |
| 팀 합류 | 인파 팀 관계가 활성화됨 | 1주 정착 확인 |
| 다시 연락 | 지금은 보류했으나 재연락하기로 함 | 다음 확인일 지정 |
| 종료 | 연락 종료 또는 후보 요청으로 마침 | 보관기간에 따라 삭제 |

후보 카드에는 `다음 행동` 한 개와 `다음 확인일` 한 개만 둔다. 첫 버전에는 자유 메모, 통화 녹음, 면접 평가, 후보 점수, 예상 수익을 넣지 않는다.

### 6.4 중복 신청과 리더 선택

- 동일 연락처는 서비스 전체에서 중복 여부를 확인하되, 다른 리더가 보유한 후보 상세정보는 노출하지 않는다.
- 후보가 새 링크에서 다시 신청하면 이미 진행 중인 대화가 있음을 후보 본인에게만 알린다.
- 후보가 현재 리더를 유지하거나 새 리더로 변경하도록 직접 선택한다.
- 리더 변경 시 이전 리더에게는 `후보가 다른 담당자를 선택해 대화가 종료되었어요`라는 상태만 보이고, 새 리더의 신원이나 상담 내용은 보이지 않는다.
- 변경 이력을 기록해 운영자가 분쟁을 확인할 수 있게 한다.

### 6.5 팀 합류와 정착 지원

1. 후보가 위촉 준비 단계에 도달하면 기존 인파 팀 초대 링크를 보낸다.
2. 후보가 가입 또는 로그인 후 팀 초대를 수락한다.
3. 후보와 인파 계정을 연결하고 후보 단계가 `팀 합류`로 바뀐다.
4. 초대한 사용자가 처음 팀원을 얻은 경우 Manager로 자동 승격한다.
5. 합류일을 기준으로 1·4·8·13주 정착 확인 일정을 만든다.
6. 각 일정에 상태, 막힘 이유, 다음 지원을 30초 안에 선택한다.

## 7. 정착 지원 설계

별도 교육 관리 시스템이나 주간 일지를 만들지 않는다. `정착 지원` 탭에서 네 번의 짧은 확인만 제공한다.

### 7.1 확인 시점

- 1주: 첫 활동 시작과 기본 도구 적응
- 4주: 첫 달 활동 유지와 초기 막힘
- 8주: 반복 활동 정착과 지원 필요 여부
- 13주: 초기 정착 완료 또는 추가 지원 판단

### 7.2 입력 항목

| 항목 | 선택값 |
|---|---|
| 현재 상태 | `활동 중`, `지원 필요`, `중단` |
| 막힘 이유 | `고객 발굴`, `상담 준비`, `상품 이해`, `업무 도구`, `시간 관리`, `조직 적응`, `개인 사정`, `해당 없음` |
| 다음 지원 | `함께 상담 준비`, `교육 연결`, `활동 계획 확인`, `도구 사용 도움`, `리더 면담`, `다음 확인만 예약`, `종료 처리` |

- 자유문구는 첫 버전에서 받지 않는다.
- 건강, 가족, 경제 상황 같은 민감한 사유를 기록하지 않는다.
- 팀원 평가점수, 순위, 수당, 보험료, 개인 매출을 정착 상태와 연결하지 않는다.
- 예정일이 되면 리더에게 인앱 알림을 보내고, 완료할 때 다음 확인일을 자동 계산한다.

## 8. 화면 구조

### 8.1 설계사 영입 홈

상단에는 오늘 해야 할 일을 먼저 보여준다.

- 24시간 안에 연락할 새 신청
- 확인일이 지난 후보
- 이번 주 대화·면담
- 이번 주 정착 확인
- 개인 영입 링크 복사

후보 단계별 숫자는 보조 정보로 두고, 사용자가 바로 행동할 수 있는 카드가 먼저 보인다.

### 8.2 영입 현황

- 데스크톱: 단계별 보기와 목록 보기
- 모바일: 다음 행동일이 빠른 순서의 세로 카드
- 검색: 후보 이름과 본인에게 공개된 연락처만
- 필터: 담당자, 단계, 출처, 다음 확인일, 경력 구분
- 빠른 행동: 전화, 문자 작성, 일정 만들기, 단계 변경
- 빈 화면: 개인 영입 페이지 만들기와 첫 링크 보내기를 안내
- 오류 화면: 저장된 입력을 보존하고 다시 시도할 행동을 제공

### 8.3 나의 영입 페이지

- 리더 사진, 이름, 소속, 직책
- 실제 활동 지역
- 팀이 제공하는 지원 방식
- 대화 진행 순서
- 자주 묻는 질문
- 경력 설계사에게 맞는 승인 문구 미리보기
- 공개·일시정지, 링크 복사, QR 저장

수익, 수수료, 정착지원금, 성공 사례 숫자는 운영 검토 없이 자유 입력할 수 없다.

### 8.4 캠페인 링크

- 개인 소개, Threads, TikTok, Instagram, 사람인, 잡코리아, 기타 채널
- 채널과 캠페인명을 선택해 별도 링크 생성
- 방문, 신청, 첫 연락, 대화·면담, 팀 합류를 집계
- 링크 일시정지와 폐기
- 원본 영입 페이지가 바뀌면 기존 활성 링크에도 같은 내용 반영

### 8.5 정착 지원

- 이번 주 확인 대상
- 지원이 필요한 팀원
- 1·4·8·13주 진행 상태
- 각 확인의 세 가지 구조화 입력
- 기한이 지난 확인과 다음 지원 행동

### 8.6 공개 영입 페이지

- 모바일 우선
- 리더의 신원과 실제 소속을 먼저 표시
- 경력 설계사에게 맞는 지원 내용을 구체적으로 설명
- 예상 수입이나 성공을 보장하는 표현은 제외
- 신청 중 로딩, 중복 신청, 만료 링크, 네트워크 재시도, 신청 완료 상태 포함
- 검색 노출은 개인 토큰 페이지에서 차단하고, 3단계 인파 공개 콘텐츠만 검색 노출

### 8.7 운영자 화면

비개발자가 다음 운영을 할 수 있어야 한다.

- 승인된 영입 소개 문구와 버전 관리
- 후보 동의 문구와 버전 관리
- 후보 삭제 요청과 보관기간 처리 현황
- 공개 링크 정지와 신고 처리
- 리더 변경 이력 확인
- 채널별 방문·신청·연락·팀 합류 집계
- Manager 자동 승격 이력과 기존 구독 상태 확인
- 개인정보 상세 열람은 업무상 필요한 관리자에게만 제한

## 9. 고객과 후보의 완전 분리

이 기능의 가장 중요한 데이터 원칙이다.

### 9.1 저장 구조

- 후보는 별도 `recruiting` 영역의 모델과 테이블에 저장한다.
- 보험 가입자인 고객의 `Customer` 모델을 재사용하지 않는다.
- 후보와 고객 사이에 외래키를 만들지 않는다.
- 후보 연락처를 고객 검색, 고객 단계별 보기, 고객 통계, 보험 분석 대상에 포함하지 않는다.
- 고객 연락처를 후보 검색, 캠페인 대상, 영입 통계에 포함하지 않는다.
- 이름이나 연락처가 같아도 자동 병합하거나 복사하지 않는다.

### 9.2 API와 화면

- 후보 API는 `/api/v1/recruiting/` 아래에 둔다.
- 고객 API와 후보 API의 직렬화 형식과 권한 검사를 분리한다.
- 전역 검색을 만들더라도 `고객`과 `설계사 후보` 결과를 별도 묶음으로 표시한다.
- 후보를 고객으로 바꾸는 버튼과 자동 변환 흐름을 만들지 않는다.
- 팀 합류 후에도 과거 후보 기록은 영입 기록으로 남고 고객 레코드로 이동하지 않는다.

### 9.3 분석과 계측

- 고객 퍼널과 영입 퍼널은 서로 다른 이벤트 이름과 관리자 지표를 사용한다.
- 고객 수, 상담 수, 보험료, 보장분석 지표에 영입 후보를 더하지 않는다.
- 영입 후보 수, 연락률, 면담률, 팀 합류율, 정착 확인률에 고객을 더하지 않는다.
- 원시 이름, 전화번호, 자유문구는 분석 이벤트에 저장하지 않는다.

## 10. 권한과 개인정보

### 10.1 기본 권한

- 후보를 받은 담당 설계사는 자신의 후보 상세정보만 본다.
- 후보가 선택한 최종 리더가 현재 담당자가 된다.
- Manager는 팀 전체 단계별 인원과 지연 건수를 기본으로 본다.
- Manager의 후보 상세 열람은 본인이 담당하거나, 지원이 필요한 건으로 역할상 배정된 경우에만 허용한다.
- 담당 변경, 상세 열람, 삭제, 리더 선택 변경은 감사 이력을 남긴다.

### 10.2 최소 수집

첫 신청에서 받는 정보는 다음으로 제한한다.

- 이름
- 휴대전화
- 경력 구분
- 현재 또는 최근 소속(선택)
- 활동 지역
- 연락 가능 시간대
- 동의 버전과 동의 시각
- 유입 링크와 캠페인

생년월일, 주민등록번호, 상세 주소, 보험 계약, 고객 명단, 수수료, 소득, 건강 정보, 가족 정보, 이력서 파일은 받지 않는다.

### 10.3 보관과 삭제

- 후보가 연락 중단이나 삭제를 요청하면 진행 대상에서 즉시 제외한다.
- 팀에 합류하지 않은 종료 후보는 종료일 또는 마지막 활동일 중 늦은 날부터 180일 후 자동 삭제한다.
- 삭제 작업은 시스템이 기록한 상태와 날짜를 함께 확인하는 다중 조건으로 수행한다.
- 팀 합류 후보는 후보 신청에 필요했던 연락처와 소속 자유값을 최소화하고, 팀 관계와 정착 확인 기록만 별도 보관한다.
- 로그에는 후보 이름, 전화번호, 신청 내용이 남지 않는다.

### 10.4 동의와 공개 문구

- 후보 동의 문구는 한 파일과 버전 상수에서 관리한다.
- 수집 목적, 연락 주체, 보관기간, 삭제 요청 방법을 신청 전에 보여준다.
- 동의 내용이 실질적으로 바뀌면 버전을 올리고 새 신청부터 새 버전을 받는다.
- 인파 공개 콘텐츠에서 특정 리더에게 정보를 전달할 때 후보가 해당 리더를 선택한 기록을 저장한다.

## 11. 영입 표현의 원칙

- `고수익 보장`, `누구나 성공`, `자동 수익`, `평생 수수료` 같은 검증할 수 없는 표현을 쓰지 않는다.
- 피라미드, 다단계, 수당 계보를 제품의 중심 가치로 표현하지 않는다.
- 제품은 경력 설계사의 선택, 리더와의 대화, 초기 정착 지원에 초점을 둔다.
- GA별 수수료표, 예상 소득 계산, 수당 비교, 정착지원금 계산은 첫 출시 범위에서 제외한다.
- 후보 평가 점수, AI 적합도, 합격·탈락 순위는 만들지 않는다.
- 사용자가 직접 수정할 수 있는 자유 홍보 문구는 제한하고, 승인된 쉬운 문장을 조합하게 한다.

## 12. 알림과 예외 처리

### 12.1 알림

- 새 대화 신청
- 24시간 안에 첫 연락
- 다음 확인일 도래
- 대화·면담 일정
- 팀 초대 수락과 Manager 승격
- 1·4·8·13주 정착 확인
- 후보의 연락 중단 또는 리더 변경

자동 문자나 카카오톡 발송은 첫 출시에서 제공하지 않는다. 설계사가 내용을 확인하고 직접 연락하도록 전화·문자 작성 바로가기를 제공한다.

### 12.2 예외 상태

- 만료·정지 링크: 현재 담당자에게 다른 연락 방법을 요청할 수 있는 안내
- 중복 신청: 후보가 진행 중인 담당자를 유지하거나 변경하는 선택
- 팀 초대 충돌: 이미 다른 팀에 속해 있으면 기존 관계를 후보가 확인한 뒤 변경
- 저장 실패: 입력값을 화면에 유지하고 다시 저장
- 네트워크 지연: 중복 제출을 막고 처리 상태 확인
- 삭제 요청 처리 중: 담당자의 연락 행동에서 즉시 제외
- Plus 만료: 팀 데이터는 유지하고 Plus 재시작 후 이어서 관리할 수 있다는 안내

## 13. 측정 지표

### 13.1 활성화 흐름

`설계사 영입 메뉴 방문 → 영입 페이지 공개 → 첫 링크 공유 → 첫 신청 → 24시간 내 첫 연락 → 대화·면담 → 팀 합류 → Manager 자동 승격`

### 13.2 핵심 지표

- 영입 페이지 공개율
- 링크 공유 후 첫 신청까지 걸린 시간
- 방문 대비 대화 신청률
- 신청 후 24시간 내 첫 행동률
- 신청 대비 대화·면담 전환율
- 대화·면담 대비 팀 합류율
- 1·4·8·13주 확인 완료율
- 4주·8주·13주 활동 중 비율
- Manager 자동 승격 성공·실패 건수
- Plus for 설계사에서 Plus for Manager로 바뀐 사용자 수

### 13.3 분석 데이터 원칙

- UTM과 캠페인 식별자를 첫 방문부터 보존한다.
- 후보의 이름, 전화번호, 소속 원문은 분석 이벤트에 넣지 않는다.
- Manager 화면에는 팀원의 개별 수입이나 고객정보가 아닌 영입·정착 집계만 보여준다.
- 인파 운영 화면에서 채널, 캠페인, 리더별 전환을 보되 개인정보 상세 접근은 분리한다.

## 16. Persona Council 결정 기록

### 16.1 참여 관점

대표·사업, 제품기획, 설계사·Manager 현장, UX, 프론트엔드, 데이터·백엔드, 보안·서버, 개인정보·준법, 브랜드, 콘텐츠, 퍼포먼스 마케팅, 그로스, QA, CS, 재무 관점으로 검토했다.

### 16.2 1라운드: 독립 제안

- 사업·제품 관점은 기존 팀 초대와 Manager 현황을 활용할 수 있는 A를 가장 빠른 검증안으로 평가했다.
- 리스크·현장 관점은 후보 연락과 초기 이탈을 함께 해결하는 B가 장기적인 유료 가치와 차별성이 높다고 평가했다.
- C는 가능성이 크지만 인파가 소프트웨어에서 모집·매칭 사업자로 바뀌는 별도 사업에 가깝다는 의견이 일치했다.

### 16.3 2라운드: 공격과 방어

- A에 대한 공격: 후보 주소록으로 끝나면 재사용과 과금 이유가 약하다.
- A의 방어: 8주 안에 링크, 신청, 연락, 면담을 가장 명확히 검증할 수 있다.
- B에 대한 공격: 정착 입력이 보고 업무와 감시로 느껴지면 사용되지 않는다.
- B의 방어: 별도 일지 없이 1·4·8·13주 네 번, 30초 선택형으로 제한하면 부담을 통제할 수 있다.
- C에 대한 공격: 복수 GA 매칭, 콘텐츠, 광고, 교육, 개인정보 제공을 동시에 운영해야 한다.
- C의 방어: 후보풀과 GA 네트워크가 형성되면 장기 해자가 크다.

### 16.4 3라운드: 최종 절충

- B를 선택하되, 정착 기능을 별도 시스템으로 확장하지 않는다.
- A의 단순한 두 화면 구조를 유지하고, 팀 합류 후 같은 흐름에 네 번의 최소 확인만 붙인다.
- C의 공개 콘텐츠 유입은 출시 3으로 분리하고, 복수 GA 자동 매칭은 제외한다.
- Manager는 추가 구매 상품이 아니라 Plus 사용자가 첫 팀원을 얻을 때 자동으로 얻는 역할과 기능으로 재정의한다.

## 17. 최종 수용 기준

다음 문장이 모두 참이어야 이 설계가 구현된 것으로 본다.

1. 설계사는 고객 메뉴를 거치지 않고 설계사 영입 업무를 시작할 수 있다.
2. 보험 가입자인 고객과 설계사 후보는 같은 이름·전화번호여도 합쳐지지 않는다.
3. 경력 설계사 후보가 자신의 최종 리더를 선택할 수 있다.
4. 모든 Plus 설계사는 영입을 시작할 수 있다.
5. 첫 팀원이 실제 팀 초대를 수락하면 같은 가격으로 Manager가 된다.
6. 자동 승격이 결제일, 쿠폰, 구독기간, 사용량을 바꾸지 않는다.
7. 기존 Manager 가입자는 이용 기록과 권한을 잃지 않는다.
8. 리더는 오늘 연락할 후보와 지원할 팀원을 한 화면에서 찾을 수 있다.
9. 정착 확인은 1·4·8·13주 네 번이며 한 번에 30초 안에 끝난다.
10. 수당 계보, 후보 점수, 자동 대량 연락, 고객정보 연결이 첫 출시 범위에 없다.
