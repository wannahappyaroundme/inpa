# 문서 06 — 담보(보장) 분류 체계 레퍼런스 & 분석엔진 '틀'

> **출처**: PM 제공(2026-06-17). 가입설계서·보험증권에 실제로 찍히는 담보명 기준. 보험사는 상품 개정·단종·신규를 수시로 하므로 정확한 최신 담보명은 **생명보험협회(klia.or.kr)·손해보험협회(knia.or.kr) 공시실**에서 확인·갱신.
>
> **이 문서의 의미**: 지금 Foliio 분석은 우리가 정한 **7-factor(7개 보장 카테고리)** 수준만 보여준다. 그러나 설계사가 고객에게 보여줘야 하는 건 **아래 전체 담보 체계 100개+**가 *한눈에 들어오는* 분석·영업·시각화 자료다. 이 문서가 그 **'틀(framework)'** 이자 OCR/AI가 인식·정규화해야 할 **사전(dictionary)** 이다.

---

## 1. 왜 이게 게임체인저인가 (PM 핵심)

### 1-1. "설계사는 비교/분석을 직접 하지만, 틀을 가진 사람이 거의 없다"
- 대부분의 설계사가 엑셀·수기로 보장분석을 *직접* 한다. 그러나 **표준화된 틀이 없어** 매번 누락·중복·들쭉날쭉.
- → Foliio가 **전체 담보 체계라는 '틀'** 을 기본 제공하면, 설계사는 "빈칸 채우기"만 하면 전문가급 분석이 완성된다. **이게 범용 AI·캔바·엑셀이 못 주는 진짜 해자.**

### 1-2. 7-factor → 전체 담보 체계로 확장
| | 현재(7-factor) | 목표(전체 체계) |
|---|---|---|
| 보여주는 것 | 7개 보장 카테고리 요약 | **15+ 카테고리 / 100+ 개별 담보** |
| 형태 | 레이더·요약 | **한눈 분석표 + 카테고리별 충족/공백 히트맵 + 영업용 시각화** |
| 가치 | "대충 어느 영역이 약하다" | "어느 *담보*가 비었고, 보험사별로 어떻게 다른지"까지 |

### 1-3. 보험사별 담보명 정규화 = 분석/OCR의 심장
같은 보장도 보험사마다 이름이 다르다:

| 표준 담보(우리 틀) | 삼성생명 | 교보생명 | 한화생명 | 신한라이프 |
|---|---|---|---|---|
| 일반암 진단비 | 암진단**급부금** | 암진단**보험금** | 암진단급부금 | 암진단보험금 |
| 뇌졸중 진단비 | 뇌졸중진단급부금 | 뇌졸중진단보험금 | 뇌졸중진단급부금 | 뇌졸중진단보험금 |

- 손해보험도 동일: 같은 담보를 삼성화재=*~보험금*, 현대해상=*~급부금* 식으로 표기.
- → **OCR이 증권에서 "암진단급부금"을 읽으면 → 표준 담보 "일반암 진단비"로 정규화**해야 비교/분석/갈아타기(승환) 비교안내서가 성립. §4가 그 정규화 사전의 seed.

### 1-4. 제품 함의 (요약)
- **분석엔진**: 7-factor → 전체 담보 트리(§2·§3)로 데이터 모델 확장. 카테고리 = 대분류, 담보 = leaf.
- **정규화 엔진**: 보험사별 담보명 → 표준 담보 매핑 사전(§4). OCR(`claude_parser`) 후처리에 결합.
- **시각화**: 카테고리별 충족률 + 담보 단위 공백 히트맵 + 갈아타기 비교표. "한눈에" 가 핵심 UX 요구.
- **틀 제공**: 이 전체 체계를 기본 템플릿으로 깔아 설계사의 "틀 부재" 문제를 해결.

## 3. 손해보험 담보 분류 체계

> 손해보험은 생명보험보다 담보가 훨씬 세분화(특히 치료비·실손·상해·배상·재물). **장기인보험(건강·암·실손)** 갈아타기의 복잡도가 여기서 나온다 = AI 가치 최대.

### [사망]

### [진단비 - 뇌혈관]

## 5. 분석엔진 · 시각화 설계 함의

### 5-1. 데이터 모델 확장
- 현 `Category` 계층 + `CustomerInsuranceDetail` → **표준 담보 트리(15+ 카테고리 / 100+ leaf)** 로 확장.
- 각 leaf에: `표준담보ID`, `카테고리`, `보험사별 alias[]`, `보장유형(진단/치료/사망/장해/생존/배상/재물)`, `지급방식(일시금/일당/실손/정액)`.

### 5-2. 정규화 파이프라인
```
증권 PDF → claude_parser OCR → 보험사별 담보명 추출
  → 정규화 사전(§4) 매핑 → 표준 담보 ID
  → 매칭 실패: fuzzy(3글자+) → 그래도 실패: 수동 보정 행
  → 8케이스 보험료 계산 + 카테고리 집계
```
- 기존 "진단/치료 키워드 과매칭" 트랩(§CLAUDE.md 8) 회피: **표준 담보 단위 정확 매핑**으로 대체.

### 5-3. "한눈에" 시각화 3종 (영업 자료)
1. **전체 담보 충족 히트맵** — 카테고리 × 담보 격자, 보유=색/공백=회색. 고객이 한 장으로 "내가 뭐가 비었는지" 인지.
2. **카테고리별 충족률 바/레이더** — 현 7-factor의 확장판(전체 체계 기준 %).
3. **갈아타기 비교표** — 기존 vs 제안을 *표준 담보 단위*로 정렬(보험사 달라도 정규화로 1:1 대응) → 승환 비교안내서(문서 05 §2)와 직결.

### 5-4. '틀' 제공 = 온보딩/콜드스타트 해결
- 신규 설계사가 빈 화면이 아니라 **전체 담보 체계 템플릿**으로 시작 → "빈칸 채우기"만으로 전문가급 분석. (문서 05의 신입 콜드스타트 대응과 연결.)

## 6. 한눈 요약
> 설계사는 분석을 *직접* 하지만 **표준 틀이 없다.** Foliio가 **전체 담보 체계(§2·§3)를 기본 틀로 깔고, 보험사별 담보명을 정규화(§4)해**, 7-factor가 아니라 **100개+ 담보가 한눈에 보이는 분석·영업·시각화 자료(§5)** 를 만든다. 이 정규화된 표준 담보 단위가 곧 **갈아타기 비교안내서(문서 05)의 1:1 대응 기반**이다. 사전은 협회 공시로 정기 갱신한다.
