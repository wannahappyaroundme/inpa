# 문서 3: AI로 영업지원을 어떻게 하는가 — 기능별 입력→처리→출력 심층 설계

> Foliio의 비전: 보험설계사가 "고객을 만나기 전 3분, 만난 후 1분, 그리고 잊지 않아야 할 모든 순간"을 AI가 대신 준비해주는 한국형 영업지원 운영체제. 본 문서는 각 AI 기능을 **입력 → Claude 처리(프롬프트 컨셉) → 출력**의 3단으로 분해하고, 모든 기능에 **보험광고 심의 가드레일 + PII 국외이전 가드레일**을 어떻게 거는지를 명시한다.

---

## 0. 읽는 법 (PM용 안내)

이 문서는 코드를 읽지 않아도 "각 기능이 무엇을 받아 무엇을 내놓는가"가 한눈에 그려지도록 설계했다. 핵심 원칙 세 가지를 먼저 박아둔다.

| 원칙 | 의미 | 왜 중요한가 |
|---|---|---|
| **분석은 끝점이 아니라 시작점** | 기존 OCR→8케이스→공유링크 파이프라인을 "자료 생산"이 아니라 "접촉 트리거"로 재배치 | 같은 엔진을 재활용하므로 신규 개발 90% 절감 |
| **모든 AI 텍스트는 단일 게이트 통과** | `core/ai_guardrail.py` 하나를 반드시 거쳐야 화면에 노출 | 보험업법 광고심의 리스크를 한 곳에서 통제 |
| **AI는 초안, 최종책임은 설계사** | "심의 안전 배지" 금지 → "1차 보조 + 면책 UI"로 포지셔닝 | 보증책임 떠안으면 Foliio가 소송 대상이 됨 |

각 기능 카드의 구조:
- **입력(Input)**: AI에 들어가는 데이터. 기존 자산 / 신규 수집을 구분.
- **처리(Claude 프롬프트 컨셉)**: 어떤 모델·어떤 system 프롬프트·어떤 출력 강제를 거는가.
- **출력(Output)**: 화면에 무엇이 뜨고 설계사가 무엇을 하는가.
- **가드레일**: 광고심의·PII를 이 기능에 어떻게 거는가.

## 1. Claude API 활용 아키텍처 (전 기능 공통 토대)

개별 기능을 설명하기 전에, 모든 AI 호출이 공유하는 공통 골격을 먼저 정의한다. 기존 `ai_summary.py` / `claude_parser.py`를 재활용하되, 그 위에 **모델 라우팅 / 가드레일 미들웨어 / 크레딧 게이팅**을 얹는다.

### 1.1 모델 라우팅 — 작업 성격별로 모델을 나눈다

비용과 품질을 동시에 잡기 위해, 작업을 세 등급으로 나누고 모델을 매핑한다. (현 코드베이스는 `claude-sonnet-4-20250514`를 OCR에 쓰고 있으나, 신규 영업지원 기능은 아래 라우팅을 권장한다.)

| 작업 등급 | 대표 기능 | 권장 모델 | 이유 |
|---|---|---|---|
| **고지능 (판단/내러티브)** | 보장공백 진단, 상담 내러티브, 비교 매트릭스 해석 | `claude-opus-4-8` | 표준 보장 기준선 대비 판정·설득 논리는 추론 품질이 곧 제품 가치 |
| **정형 대량 (멘트/메시지)** | 카톡 메시지 3종 초안, 만기/생일 멘트 | `claude-haiku-4-5` | 톤만 바꾼 반복 생성 — 빠르고 저렴 |
| **경계 검열 (가드레일 2단계)** | 광고심의 경계사례 warn 판정 | `claude-haiku-4-5` | 짧은 입력·이진 판정, 지연 최소화 |
| **OCR 파싱 (구조 추출)** | 증권 다건 일괄 파싱 | `claude-opus-4-8` (스키마 강제) 또는 현 Sonnet 유지 | 보험사별 양식 차이 흡수에 추론 필요 |

> **모델 ID 주의**: 위 ID 문자열은 그대로 사용하며 날짜 접미사를 붙이지 않는다(`claude-opus-4-8`이 완전한 ID). `claude-opus-4-8`/`claude-haiku-4-5`는 모두 1M/200K 컨텍스트로 본 용도에 충분.

### 1.2 공통 호출 패턴 — prompt caching + 출력 강제 + adaptive thinking

세 가지 공통 장치를 모든 호출에 표준 적용한다.

1. **Prompt caching (비용 절감의 핵심)**
   - 표준 보장 기준선, 광고심의 룰셋, 톤 가이드 등 **변하지 않는 거대 system 프롬프트**를 캐시 prefix로 고정.
   - 캐시는 **prefix match** — prefix 안에 `datetime.now()`·고객 ID 같은 가변값을 절대 넣지 않는다. 고객별 데이터는 항상 messages 끝(마지막 breakpoint 뒤)에 배치.
   - 효과: 같은 기준선을 N명 고객에게 반복 호출 시 캐시 read는 기본 입력가의 ~10% 수준.

2. **출력 강제 (`output_config.format` = json_schema)**
   - 진단·메시지·파싱 결과를 항상 고정 스키마로 받는다. FE가 깨지지 않고, 가드레일이 필드 단위로 검사 가능.
   - (구) `output_format` 파라미터가 아니라 `output_config: {format: {...}}` 사용. 마지막 assistant prefill은 4.6+ 모델에서 400이므로 **절대 사용 금지** — 출력 강제는 전부 스키마로.

3. **adaptive thinking + effort**
   - 고지능 작업: `thinking: {type: "adaptive"}` + `output_config: {effort: "high"}`.
   - 정형 대량: effort `low`~`medium`로 비용 절감.
   - 사용자에게 추론을 노출할 필요가 없으므로 display는 기본값(omitted) 유지.

### 1.3 야간 배치 — 리텐션 멘트는 Batches API로 50% 절감

만기·생일 워치독은 실시간성이 필요 없다. cron이 야간에 대상 고객을 추출한 뒤 **Message Batches API**(`/v1/messages/batches`)로 일괄 처리하면 토큰 비용 50% 절감. 결과는 다음날 아침 9시 알림 큐에 적재.

### 1.4 호출 흐름 한 장 요약

```
[기능 진입]
   │
   ├─ 기존 자산 로드 (calculate_analysis JSON / OCR 텍스트 / achievement_data)
   │
   ▼
[Claude 호출]  ── model 라우팅 + cached system(기준선·룰셋·톤) + output_config.format(스키마)
   │
   ▼
[ai_guardrail.py 단일 게이트]  ── ① 정규식 block  ② Haiku warn  ③ 면책 UI
   │
   ▼
[설계사 1탭 검수·편집]  ──→  클립보드 복사 / 알림 큐 / 화면 카드(상담용 한정)
   │
   ▼
[크레딧 차감 + 측정 이벤트(발송수·열람률·귀속)]
```

### 1.5 재활용 자산 vs 신규 개발 (개발량 가늠)

| 기존 자산 (재활용) | 위치 | 신규로 얹는 것 |
|---|---|---|
| `claude_parser.py` (OCR) | `core/ocr/` | 멀티 PDF 일괄 UI + 스키마 강화 |
| `ai_summary.py` (Claude 요약) | `core/` | 목적별 프롬프트 분기(멘트/진단/내러티브) |
| `calculate_analysis()` / `compare` | `customers/calculate.py` | 출력 JSON을 프롬프트 입력으로 매핑 |
| `content_filter.py` (PII 정규식) | `community/` | 광고심의 금칙어 룰셋으로 확장 |
| `credit.py` (크레딧) | `membership/` | AI 호출 차감 연결 |
| SENS/NHN/Notification/cron | 각 앱 | 트리거 추출 + 할일 큐 |

## 2. 보험 컴플라이언스 가드레일 (먼저 만들어야 하는 위생 미들웨어)

> **이 기능이 없으면 다른 AI 기능 전부 출시 불가.** 해자가 아니라 "최소 위생 + 면책"이다. MVP 1순위.

### 2.1 3중 구조 — 결정적 차단 + LLM warn + 사람 승인

```
임의 AI 텍스트
   │
   ▼
1단계 ── 결정적 정규식 (deterministic)   → BLOCK 권한은 여기만 가짐
   │     · 금칙어("100% 보장", "최고", "무조건")
   │     · 보장금액 단정 패턴("암 진단비 X만원 부족")
   │     · 경품/특별이익("첫 달 대납", "사비로")
   ▼
2단계 ── Haiku 경계 검열 (probabilistic)  → WARN만 가능, BLOCK 권한 없음
   │     · 모호한 비교 표현, 단정적 권유 뉘앙스
   │     · "왜 위험한지 + 안전 대안" 툴팁 생성
   ▼
3단계 ── 사람 최종승인 (설계사)            → 면책 동의 후 발송
```

**핵심 설계 결정**: LLM verdict를 자동 신뢰하지 않는다. Haiku는 막을 권한이 없고 경고만 단다. 막는 것은 오직 결정적 정규식 — 오탐으로 정상 멘트가 차단되는 사고를 통제 가능하게 유지한다.

### 2.2 입력 → 처리 → 출력

| 단계 | 내용 |
|---|---|
| **입력** | 임의 AI 생성 텍스트(멘트/진단/내러티브) + 컨텍스트(채널, 대외배포 여부) |
| **처리 (1단계)** | `content_filter.py` 정규식 구조 재활용 + 보험업법 금지표현 JSON 룰셋. block은 정규식만. |
| **처리 (2단계)** | Haiku, cached system에 보험업법 광고심의 요지 + 안전대안 예시 고정. 출력 스키마: `{verdict: "ok"|"warn", reason, safe_alternative}`. effort `low`. |
| **출력** | block → 빨간 차단 + 사유. warn → 노란 툴팁("왜 위험한지 + 안전 대안") + 설계사 확인 토글. ok → 통과. **모든 경우 하단에 면책 카피 고정.** |

### 2.3 이 기능의 한계를 명시하라 (리스크 직시)

필터가 **못 잡는** 것을 UI와 약관에 명시한다. 이게 면책의 핵심.

- **비교광고**("타사 대비 2배"), **특별이익**("사비로 첫 달 대납"), **불완전판매** 같은 **맥락 위반**은 정규식·LLM 모두 못 잡는다.
- 따라서 "심의 안전 배지"를 붙이는 순간 Foliio가 보증책임을 떠안아 소송 사유가 됨 → **절대 금지**.
- 포지셔닝 카피: *"1차 실수 방지 보조입니다. 최종 검토·발송 책임은 설계사 본인에게 있습니다."*

### 2.4 품질 보증

- **위반 카피 100건 골든 회귀 테스트 필수**: 룰셋 변경 시마다 100건 통과/차단 결과를 검증. `content_filter.py` 테스트 패턴 재활용.
- 룰셋은 **명시 리스트로 확장**하되 정규식을 느슨하게 풀지 않는다(진단비 OCR 과매칭 트랩과 동일 교훈).

**재활용**: `community/content_filter.py` 정규식 + `profanity_words.json` 구조. **신규**: 보험업법 금지표현 JSON 룰셋 + Haiku 검열 프롬프트 + 공유 미들웨어 + 면책 UI.

## 3. PII 동의·민감정보 국외이전 가드레일 (모든 AI의 선결 과제)

> 고객 병력=민감정보를 Claude API(Anthropic, 미국)로 전송. **개인정보보호법상 민감정보 국외이전 별도 동의가 없으면 AI 기능 전체 출시 불가.** MVP 1순위, 법무 선결.

### 3.1 가드레일 구성 (AI 처리 아님 — 동의·로그 인프라)

| 항목 | 내용 |
|---|---|
| **동의 항목 확장** | `customer-agree` 동의서에 "AI 분석을 위한 개인(민감)정보 국외 제3자 제공" 항목 추가. Anthropic(미국)을 수탁자로 명시. (랜딩 privacy 페이지의 "Anthropic, Inc. (US)" 표기와 일관 유지) |
| **동의 로그** | 1탭 구두동의 + 타임스탬프 로그. 누가 언제 동의했는지 감사 추적. |
| **share_token 보호** | 만료/회수/조회로그/robots 차단. 민감 분석이 단톡방·검색엔진에 영구 노출되는 사고 방지. |
| **제3자 수집 분리** | 바이럴 루프의 지인 B 셀프입력은 **제3자 수집 소지** → 별도 동의 동선. |

### 3.2 각 AI 기능에 거는 방식 (게이팅)

모든 AI 기능 진입 전, 해당 고객의 동의 상태를 검사한다.

```
AI 기능 진입
   │
   ▼
customer-agree 동의 상태 체크
   ├─ 국외이전 동의 O → AI 호출 허용
   └─ 동의 X → 1탭 동의 모달 → 동의 후 진행 / 미동의 시 AI 기능 비활성(분석만 가능)
```

### 3.3 마스킹 옵션 (법무 결과에 따라)

법무가 "병력 원문 전송 불가"로 판단할 경우의 대비책: **전송 전 마스킹**. 진단명을 카테고리 코드로 치환(예: 특정 암종명 → "암 진단 이력 있음")해 식별성을 낮춘 뒤 Claude에 전송. 진단 기능은 카테고리 단위로 동작하므로 정밀도 손실 최소.

**열린 질문(Open Question)**: 현 `customer-agree`로 커버 가능한가, 별도 동의서·온프레미스·마스킹 후 전송이 필요한가? → **법무 검토 결과가 AI 기능 전체의 출시 가부를 결정.**

## 4. 분석-콘텐츠 파이프라인 기능군

### 4.1 사전준비형 원클릭 니즈환기 (Analysis-to-Kakao) — MVP

미팅 전 증권 취합·보장분석에 30분~1시간. 분석이 설계사 혼자 보는 자료로 끝나고 고객 접촉으로 연결 안 됨.

| 단계 | 내용 |
|---|---|
| **입력** | `calculate_analysis()` JSON(8케이스 보험료 집계) + 고객 메타(직업위험/연령/성별/병력) + `share_token` |
| **처리** | `claude-opus-4-8`. system(cached)=표준 보장 기준선 + 카톡 멘트 톤 가이드. user=분석 JSON. **출력 스키마**: `{gaps:[...], messages:{soft, numeric, urgent}}` — 톤 3종(부드러움/숫자강조/긴급) 강제. 멘트 안에 share_token 링크 자동 삽입. |
| **출력** | 진단 카드 + 카톡 멘트 3종 → 설계사 1탭 검수·편집 → **클립보드 복사 + 카톡 앱 열기** |
| **포지셔닝** | "라이브 현장"이 아니라 **"미팅 전날 사전 준비"** — OCR 오류·지연 리스크 회피 |

**프롬프트 컨셉 (system, 캐시 고정)**:
> "당신은 보험설계사의 고객 응대 멘트 초안 작성 보조다. 입력된 보장분석 결과에서 공백을 찾아, 고객이 읽을 카톡 멘트를 3가지 톤으로 작성하라. 보장금액을 단정하지 말고 '검토가 필요해 보인다' 수준으로 표현하라. 타사 비교·수익률·100%/최고 같은 표현을 절대 쓰지 마라. 각 멘트 끝에 `{share_link}` 자리표시자를 둬라."

## 5. AI 카톡 메시지 엔진 (목적 enum + 가드레일) — MVP

> 한국 영업 90%가 카톡. 5개 페르소나가 같은 기능을 5번 재발명 → **단일 엔진으로 통합.** 리텐션·소개·온보딩이 전부 이 엔진을 목적 파라미터만 바꿔 호출.

| 단계 | 내용 |
|---|---|
| **입력** | 고객 컨텍스트(메타) + **목적 enum**: `첫인사 / 안부 / 약속 / 생일 / 만기 / 소개요청 / 거절후속` |
| **처리** | `claude-haiku-4-5`. system(cached)=목적별 톤 가이드 + 금칙어. user=고객 메타 + 선택 목적. **출력 스키마**: `{drafts: {tone_a, tone_b, tone_c}}`. |
| **출력** | 초안 3종 → 검수·편집 → **클립보드 복사 + 카톡 앱 열기** |

**정직한 발송 격하 (리스크 직시)**: `Kakao.Share.sendDefault`는 자동/1:1 발송 불가(수동 공유 시트). **"원탭 자동발송"은 거짓 전제.** 개별 멘트는 클립보드 복사 기본. 대량 발송은 §8 알림톡 사전심의 템플릿 경로로 분리.

**거절처리 코칭/롤플레이 (목적=거절후속 확장)**:
- 입력에 고객의 거절 사유 텍스트 추가.
- Claude가 ① 거절 유형 분류 ② 재접근 멘트 3종 ③ 롤플레이 예상 반론·대응을 출력.
- **상담용 한정**(고객에게 직접 가는 멘트는 §2 통과한 재접근 멘트만).

**니즈분석 인터뷰 (목적=니즈환기 확장)**:
- 고객 메타 기반으로 Claude가 **상담 시 던질 질문 리스트**(직업위험·가족구성·기존보장 공백 탐색 질문)를 생성. 설계사 화면 전용 가이드.

**가드레일**: 모든 초안 §2 통과분만 노출. §3 동의 체크. 거절 사유에 PII 포함 가능 → `content_filter` PII 검사.

**재활용**: Claude, `content_filter` 확장, 카카오 Share, SENS, Customer 메타. **신규**: 목적 enum + 생성 프롬프트 + 메시지 모달. (5중 중복 통합)

## 9. 리드 성장 루프 — 바이럴 진단 + 리퍼럴 체인

### 9.1 바이럴 진단 루프 (공유링크 → 소개 자동귀속) — Phase 2

> 리드 고갈이 도태 1순위. 공유 분석링크가 1회성으로 끝나고, 만족 고객이 지인을 데려올 통로가 없음. **수기 입력 없이 소개 데이터 채워지는 유일한 길.**

| 단계 | 내용 |
|---|---|
| **입력** | 고객 A가 받은 share_token 분석링크 |
| **처리** | `claude-opus-4-8`가 분석 톤에 맞춘 "주변에 흔한 보장공백" **소개 유도 카피** 자동 삽입(링크 하단) |
| **출력** | 링크 하단 CTA "내 보험도 1분 무료 진단(담당: 김설계사)" → 지인 B 간이입력 → 설계사에 '소개 리드 도착' 알림 + `Customer source=referral, referred_by=A` 자동 기록 |

**가드레일 (제3자 수집 — 법무 선결)**:
- 지인 B 셀프입력은 **동의 없는 제3자 수집 소지** → B의 1탭 PII 동의 동선 필수(§3 별도 동의).
- 삽입 카피는 §2 통과(보장내용 단정 금지, 자기진단 유도만).
- share_token 만료/회수/robots 차단 미비 시 민감 분석이 단톡방·검색엔진 영구 노출.

**재활용**: share_token 공개 analysis, landing 인프라. **신규**: `Customer(source/referred_by)` 필드 + CTA 컴포넌트 + 셀프진단 랜딩 + B의 PII 동의 동선 + 귀속 로직.

### 9.2 리퍼럴 체인 추적 + 소개요청 타이밍 — Phase 2

| 단계 | 내용 |
|---|---|
| **입력** | 바이럴 루프가 채운 `referred_by` 데이터(수기 입력 의존 금지) |
| **처리 (1차 규칙)** | **규칙기반 타이밍**(계약 직후/만기갱신 성공 직후 D+7) — 'AI 타이밍 감지'는 과욕이라 단순 규칙으로 시작 |
| **처리 (멘트)** | 소개요청 멘트만 §5 메시지 엔진(목적=소개요청)으로 생성 |
| **출력** | 만족 이벤트 D+7 알림 → 소개요청 멘트 → **소개 체인 트리 시각화** |

바이럴 루프(유입)와 한 시스템의 양면, 데이터 모델 공유.

## 10. 퍼스널 브랜딩 콘텐츠 (광고심의 회피 한정)

> 보장내용·수익률·타사비교를 **원천 배제**하고 자기소개·연차·신뢰 위주 카피만 자동생성해 광고심의를 회피. **후행 활성화 — 파이프라인 신뢰가 쌓인 뒤 얹는다.**

### 10.1 성과·셀프브랜딩 자동 카피 (AI SNS 콘텐츠 엔진) — Phase 3

| 단계 | 내용 |
|---|---|
| **입력** | `achievement_data` KPI(고객수/보험수/연차/연환산보험료/카테고리분포) |
| **처리** | `claude-haiku-4-5`가 **보장내용·수익률·타사비교·경품 원천 배제**한 자기브랜딩 카피 생성, promotion 템플릿 슬롯 자동 채움 |
| **출력** | 월 1회 '이번 달 성과 카드 준비됨' → 자동 채움 미리보기 → 미세조정 → 카톡/SNS 공유 |

**가드레일 (프롬프트 레벨 원천 차단)**: system에 "보장내용·수익률·타사비교·경품을 절대 언급하지 마라. 자기소개·연차·고객수·신뢰만 다뤄라." → §2 통과는 이중 안전망. **워터마크는 설계사가 싫어하므로 강제 안 함**(추천코드 링크가 진짜 바이럴).

### 10.2 고객 후기·만족도 1탭 수집 → 후기 카드 — Phase 2

| 단계 | 내용 |
|---|---|
| **입력** | 온보딩 후 1탭 별점 설문 응답 |
| **처리** | 4★+면 `claude-opus-4-8`가 한줄평을 **PII 제거·다듬어** 인용 카드뉴스 카피로 변환 |
| **출력 (게이팅)** | 4★+ → 후기카드 생성 + 소개요청 트리거 / **1~3★ → 외부 노출 0, 설계사에 '케어 필요' 알림**(평판 리스크 게이팅) |

**가드레일**: 후기 원문에 PII 포함 가능 → `content_filter` PII 제거 필수. 변환 카피 §2 통과.

### 10.3 네이버/당근 로컬 발견 SEO 콘텐츠 — Later

| 단계 | 내용 |
|---|---|
| **입력** | 월초 콘텐츠 캘린더 주제 |

## 11. 측정·과금 거버넌스

### 11.1 AI 비용 게이팅 + North Star 대시보드 — Phase 2

| 단계 | 내용 |

**북극성 지표**: `미팅당 share_token 발송수 × 고객 열람률 × 소개 귀속 리드수`. KPI가 '분석 건수'에서 '영업 행동'으로 이동.

**비용 누수 차단**: 베타/무료 사용자가 메시지·진단 무한 호출 시 Claude 비용이 샘 → 크레딧 차감 필수(단위경제 붕괴 방지).

**열린 질문**: 발송이 클립보드 복사 기반이면 **발송 이벤트를 어떻게 정확히 계측**하는가?(클립보드 복사 버튼 클릭 = 발송 이벤트로 근사)

## 14. MVP 출시 순서 (의존성 기준)

```
[선결] §3 PII 국외이전 동의 (법무) ──┐
[선결] §2 컴플라이언스 가드레일 ─────┤── 이 둘 없으면 아래 전부 출시 불가
                                      │
[1] §4.4 일괄 OCR (데이터 입력 토대) ─┤
[2] §5 메시지 엔진 (단일 엔진 통합) ──┤
[3] §4.1 니즈환기 + §4.2 진단 ───────┤── 데일리 훅
[4] §6 만기 워치독 ──────────────────┘── 리텐션 복리(ROI 최상)
```

**핵심 메시지**: §2 가드레일과 §3 동의가 **모든 AI 기능의 공통 토대**다. 이 둘을 먼저 세우고, 그 위에 OCR→메시지→진단→리텐션 순으로 얹는다. 분석 엔진은 이미 깔려 있으므로(90% 재활용), 신규 개발은 "프롬프트 + 가드레일 미들웨어 + 할일 큐 UI"에 집중된다.
