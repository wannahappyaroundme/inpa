# Foliio 기능 카탈로그 — 영업 퍼널 단계별 정리

> **문서 목적**: Foliio를 '분석 계산기'에서 '한국형 영업지원 운영체제(영업 OS)'로 재배치하는 전체 기능을, 보험설계사의 실제 영업 퍼널(발굴→접촉→니즈분석→제안→클로징→유지갱신→소개→브랜딩) 순서대로 펼쳐 보여준다. PM이 "어떤 기능이, 어느 영업 순간에, 어떤 기존 자산 위에, 얼마나 급하게 얹히는지"를 한눈에 잡도록 설계했다.
>
> **읽는 법**:
> - 각 단계 섹션 = 표(기능 요약) + 핵심 기능 1~2문단 해설
> - **우선순위 Tier**: `MVP`(1차 출시 필수) → `Phase2` → `Phase3` → `Later`
> - **임팩트/노력**: `임팩트`(영업 성과 기여) / `노력`(개발 공수). `high/med/low`
> - **재활용 기존자산**: 이미 깔려 있어 그대로 쓰는 Foliio 자산. 신규 개발 최소화의 근거.

---

## 0. 전체 그림 — 가치 이동 한 줄 요약

| | 기존 (분석 계산기) | 전환 후 (영업 OS) |
|---|---|---|
| 분석의 위치 | **끝점** — 설계사 혼자 보는 정적 자료 | **시작점** — 접촉·후속·재활용의 트리거 |
| 분석 이후 노동 | 전부 설계사 수기 (멘트·후속·재활용) | AI 초안 + 자동 큐로 거의 무노동 |
| 핵심 KPI | 분석 건수 | 미팅당 공유링크 발송수 × 고객 열람률 × 소개 귀속 리드수 |
| 과금 기준 | (베타 무료) | 분석=무료 미끼 / **발송·자동화·귀속=유료** |

**6개 전략 기둥(Pillar)이 퍼널 전체를 가로지른다.**

- **① 분석-콘텐츠 파이프라인** (Analysis-to-Action) — 유일한 진짜 해자. OCR→8케이스→공유링크를 접촉 트리거로 재배치.
- **② 컴플라이언스 안전망** — 해자가 아니라 '위생'. 모든 AI 텍스트의 단일 게이트 + 면책.
- **③ 리텐션 엔진** — 만기·생일 복리 구간. 자동 트리거 + 할일 큐.
- **④ 리드 성장 루프** — 공유링크를 신규 리드로 되돌리는 바이럴 귀속.
- **⑤ 퍼스널 브랜딩 콘텐츠** — 광고심의 회피 한정. 후행 활성화.
- **⑥ 측정·과금 거버넌스** — North Star + AI 비용 게이팅.

## 1. 발굴 (Lead Generation)

> "만날 고객이 없다"가 한국 설계사 도태 1순위 원인. 이 단계의 기능들은 **수기 입력 없이** 리드·진척·시드를 자동으로 채우는 데 집중한다. (입력 노동에 의존하는 기능은 3주 만에 버려진다는 가정이 깔려 있다.)

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| 바이럴 진단 루프 (공유링크→소개 자동귀속) | 리드 고갈. 공유 분석링크가 1회성으로 끝남 | 분석 톤에 맞춘 소개 유도 카피 자동 삽입 (Sonnet) | Phase2 | share_token 공개 analysis, Customer 모델, landing 인프라 | high / med |
| 이벤트 자동전이 파이프라인 뷰 (카드피드) | Customer가 정적 마스터라 진척이 안 보임 | 정체 딜 규칙 감지 → 넥스트액션 추천 (2차) | Phase2 | Customer 모델, Notification, 리텐션 트리거 | med / med |
| 콜드스타트 부트스트랩 + 빈 상태 aha | 신입은 고객 0·증권 0·KPI 0으로 진입해 첫 5분에 이탈 | 본인 증권 1건으로 즉시 진단 데모 (Sonnet) | Phase3 | OCR, 진단 기능, promotion | med / low |
| 설계사→설계사 B2B 팀 도입 루프 | 모든 바이럴이 '설계사→고객'뿐, '설계사→설계사' 누락 | 없음 (성장 메커니즘) | Later | 추천코드(바이럴 루프), 멤버십 | med / low |

### 바이럴 진단 루프 — 발굴 단계의 성장 엔진 (Phase2)

리드 고갈이라는 가장 근본적인 도태 원인을 정면으로 친다. 기존에는 고객에게 보낸 `share_token` 분석링크가 1회성으로 소비되고 끝났다. 여기에 링크 하단 CTA를 박는다 — **"내 보험도 1분 무료 진단 (담당: 김설계사)"**. 고객 A가 만족하면 지인 B가 이 CTA로 들어와 간이입력만 하고, 설계사에게는 '소개 리드 도착' 알림이 뜨며 `Customer.source=referral`, `referred_by=A`가 **자동 기록**된다. 수기 입력 없이 소개 데이터가 채워지는 **유일한 길**이라는 점이 핵심이다.

재활용 비중이 높다(공개 analysis API + landing 인프라). 신규 개발은 CTA 컴포넌트 + 셀프진단 랜딩 + **지인 B의 PII 동의 동선(법무 선결)** + 귀속 로직이다. 동의 동선이 빠지면 '동의 없는 제3자 수집' 리스크로 출시 자체가 막히므로, PII 동의 기능(전 단계 공통)과 묶어서 봐야 한다.

### 이벤트 자동전이 파이프라인 뷰 — 칸반이 아니다 (Phase2)

진척이 안 보이는 문제를 풀되, **수기 입력 칸반은 명시적으로 폐기**한다(5개 페르소나 합의: 끌어올리는 칸반은 3주면 버려진다). 1차는 비AI로, `lead_stage`가 발송/열람/만기 이벤트로 **자동 전이**된다. 활동로그는 발송·열람·만기를 자동 캡처(수기 0)하고, 결과는 드래그 칸반이 아니라 모바일 **카드 피드 '오늘 할 일'**로 보여준다. 2차에서 정체 딜('D+14 미접촉')을 규칙으로 감지해 넥스트액션 1~2개를 추천한다. 리텐션 엔진과 활동로그를 공유한다.

## 2. 접촉 (Contact / Outreach)

> 한국 영업의 90%는 카톡으로 움직인다. 첫인사·안부·약속 멘트를 매번 수기로 쓰는 노동을 단일 엔진으로 흡수한다.

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| AI 카톡 메시지 엔진 (목적 enum + 가드레일) | 카톡 멘트를 매번 수기. 5개 페르소나가 같은 기능을 5번 재발명 | 목적 enum + 톤 3종 초안 생성 (Haiku), 가드레일 통과분만 노출 | MVP | Claude, content_filter 확장, 카카오 Share, SENS, Customer 메타 | high / med |

### AI 카톡 메시지 엔진 — 5중 중복을 통합한 단일 엔진 (MVP)

리텐션·소개·온보딩·생일·거절후속 등 여러 페르소나가 각자 메시지 기능을 재발명할 뻔한 것을, **하나의 엔진을 목적 파라미터만 바꿔 호출**하는 구조로 통합했다. 사용 흐름은 단순하다 — 고객 선택 → 목적 선택(첫인사/안부/약속/생일/만기/소개요청/거절후속) → Haiku가 톤 3종 초안 생성 → **가드레일 통과분만 노출** → 설계사 검수·편집 → 클립보드 복사 + 카톡 앱 열기.

**정직한 한계 명시**: 카카오 인프라상 1:1 자동발송은 불가하므로 기본값은 '복사'다. 대량 발송은 별도로 **알림톡 사전심의 템플릿 경로**로 분리한다(발송 채널 현실화 기능 참조). 이 엔진은 Plus 티어의 '매일 켜는 데일리 훅'으로, 전환의 핵심 동력이다.

## 3. 니즈분석 (Needs Analysis) — 파이프라인의 심장

> Foliio의 기존 강점(OCR→8케이스)이 가장 두껍게 쌓인 구간. 여기서 분석을 '끝점'이 아니라 '접촉 트리거'로 바꾸는 변환이 일어난다. **MVP 기능이 가장 많이 몰려 있다.**

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| 사전준비형 원클릭 니즈환기 (Analysis-to-Kakao) | 미팅 전 분석 30분~1시간, 분석이 접촉으로 연결 안 됨 | 보장공백 판정 + 카톡 멘트 3종(부드러움/숫자/긴급) (Sonnet) | MVP | OCR, calculate.py, share_token, ai_summary.py, 카카오 Share — 90% 재활용 | high / low |
| AI 보장공백 진단 + 상담 내러티브 | '왜 공백인지·어떻게 설명할지'를 매번 수기, 신입은 해석 불가 | 분석+고객메타 → 공백 구조화 {category, current, recommended_band, severity, talk_track} (Sonnet) | MVP | calculate.py, ai_summary.py, Customer 직업위험/병력/성별 필드, 분석탭 UI | high / med |
| 증권 다건 일괄 OCR + 보장 매핑 정확도 강화 | 증권 취합 수기, 진단비/처치/치료 오매칭·double-count 트랩 | 보험사별 양식 흡수, 스키마 강제(output_config.format), 신뢰도 낮으면 수동 행 폴백 | MVP | claude_parser, extract_text_from_pdf, CustomerInsurance/Detail, 8케이스, 크레딧 차감 | med / low |

### 사전준비형 원클릭 니즈환기 (Analysis-to-Kakao) — 가장 가성비 높은 MVP (high/low)

**임팩트는 높고 노력은 낮다 — 1순위로 짚어야 할 기능.** 90%가 기존 자산 재활용이기 때문이다. 흐름은 이렇다 — (미팅 전날) 증권 업로드 → OCR 파싱 → 8케이스 계산 → **진단 카드 + 멘트 3종** → 설계사가 1탭 검수·편집 → 클립보드 복사 + 카톡 앱 열기. Sonnet이 `calculate_analysis()` JSON을 입력받아 보장공백을 판정하고, 가드레일 통과 후 `share_token` 링크를 멘트에 삽입한다.

**포지셔닝이 리스크 회피의 핵심**이다. '라이브 현장'이 아니라 **'사전 준비'**로 잡아 OCR 오류·지연 리스크를 회피한다. 신규 개발은 멘트 생성 프롬프트 + 링크 삽입 + 검수 게이트뿐. Foliio의 기존 분석 파이프라인을 '영업 행동의 시작점'으로 바꾸는 대표 기능이다.

### AI 보장공백 진단 + 상담 내러티브 — 숨은 EFFORT 폭탄 주의 (MVP)

8케이스 숫자는 이미 나오지만, '왜 공백인지·어떻게 설명할지'는 매번 수기였고 신입은 해석 자체를 못 했다. Sonnet이 분석 JSON + 고객 메타(직업위험/연령/성별/병력)를 받아 **표준 보장 기준선 대비 공백**을 구조화 출력하고, 분석탭 하단에 진단 카드 + 상담용 멘트를 띄운다. 기준선은 system 프롬프트에 고정해 prompt caching으로 비용을 절감한다.

⚠️ **숨은 공수 경고**: '신규 = 표준 보장 기준선 데이터셋'인데, 이게 단순 데이터가 아니다. **출처(금감원/보험연구원/자체)와 면책 문구 정의가 진짜 EFFORT 폭탄**이다. 기준선 권위가 불명확하면 '암 진단비 X만원 부족' 같은 단정이 **부적합 권유(오진 책임)**로 귀결된다. 그래서 진단 카드는 **설계사 화면 전용(대외배포 아님)**으로 한정하고, UI에 '참고용 초안, 최종판단은 설계사' 면책을 명시해 광고심의를 원천 회피한다. (Open Question: 기준선 출처 확정이 이 기능 출시의 선결 조건.)

### 증권 다건 일괄 OCR — 기존 트랩 정리 + 일괄화 (MVP)

기존 단건 OCR을 다건 일괄로 확장하고, 알려진 트랩(진단비/처치/치료 오매칭, 갱신/비갱신 double-count)을 정확도 강화로 잡는다. 멀티 PDF(암호화 포함) 일괄 업로드 → 추출 → Sonnet 파싱(스키마 강제, 표준 담보 1개씩만) → `CustomerInsurance/Detail` 자동생성, `portfolio_type=1` 크레딧 차감. **발송 전 1탭 검수 게이트**로 OCR 오류를 정정한다. 신규는 일괄 업로드 UI + 스키마 강화 정도로 노력이 낮다.

## 4. 제안 (Proposal)

> 기존vs제안 비교가 최대 시간소모이자 ★★★★★ 레버리지 구간. 단, **타사 비교표 자동 배포는 비교광고 규제 직격탄**이라 '상담용 한정'으로 신중하게 설계한다.

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| 비교 매트릭스 → 상담용 내러티브 (배포용 자동생성 금지) | 기존vs제안 비교가 최대 시간소모, 타사 비교 자동배포는 규제 직격 | compare에서 '핵심 변화 3가지' 추출 → 설득 논리 생성 (Sonnet) | Phase2 | compare(calculate.py), 8케이스, share_token, ai_summary, 가드레일 | med / med |
| 라이브 프레젠테이션 모드 (고급 옵션) | 폰으로 고객 앞 전문가 연출 욕구 | 없음 (LLM 카피 제외 — 라이브 지연=어색한 침묵) | Phase3 | analysis 차트 컴포넌트, compare API — 순수 Angular UI(BE 0) | med / low |

### 비교 매트릭스 → 상담용 내러티브 — '배포용 자동생성 금지'가 핵심 설계 (Phase2)

기존 `compare` 매트릭스에서 Sonnet이 '핵심 변화 3가지'를 추출해 설득 논리를 만든다. **단, 생성물은 상담용(설계사 화면 전용)으로만** — 배포용 자동 PDF는 비교광고 규제 + 부적합 권유 책임 + LLM의 픽셀 레이아웃 약점이 겹친 법적 지뢰라 **분기 자체를 삭제**한다. 배포는 반드시 사람 검수를 거친다. 신규 개발은 '상담용 내러티브 프롬프트(배포용 분기 삭제)'다.

### 라이브 프레젠테이션 모드 — 의도적으로 강등 (Phase3)

폰으로 고객 앞에서 전문가처럼 연출하려는 욕구를 푼다. **단, LLM 카피 생성은 제외**한다 — 라이브 지연이 어색한 침묵이 되어 신뢰가 즉사하기 때문이다. 이미 계산된 분석 차트를 **결정적으로 렌더만** 하고(헤더 숨김·큰 폰트·다크 배경), 비포애프터를 스와이프한다. 현장 OCR 라이브는 고위험·저빈도라 강등했고, 사전준비형이 기본이다. (Open Question: 상위 설계사의 '고객 앞 데모' 수요가 실제 차별화 가치인지 사용자 인터뷰로 확인 필요.)

## 6. 유지갱신 (Retention) — ROI 최상 구간

> 신규 획득보다 비용이 낮고 측정 가능한 **복리 구간**. 이미 보유한 만기/생일 데이터 + cron + SENS/NHN/Notification 인프라 위에 트리거·할일 큐만 얹는다.

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| 만기·갱신 자동 워치독 (리텐션 엔진) | 만기·갱신 놓쳐 실효·해지 → 수익 직격. 자동 워크플로 부재 | D-60/30/7 추출 → 8케이스 갱신 FV 재시뮬 → 멘트+변화요약 (Haiku 야간 Batches 50% 할인) → 가드레일 | MVP | Customer 만기/생일, 8케이스, process_dormancy/notifymembership cron, Notification, SENS/NHN/인앱 | high / med |
| 발송 채널 현실화 (알림톡 사전심의 + 스팸 거버넌스) | Share.sendDefault는 자동발송 불가, 알림톡은 가변카피 심사 불가, 친구톡은 수신동의·시간 규제 | 없음 (인프라) | Phase2 | kakao_biztalk.py(pending), SENS SMS, Notification | med / high |

### 만기·갱신 자동 워치독 — 측정 가능한 ROI 최상 (MVP)

만기·갱신을 놓치면 실효·해지로 설계사 수익이 직격당하는데, 데이터는 이미 있지만 자동 워크플로가 없었다. daily cron(기존 `expirememberships` 패턴 재활용)이 D-60/30/7 만기·생일 임박 고객을 추출 → **8케이스 갱신 FV를 재시뮬**해 '왜 갱신해야 하는지'를 데이터로 설득 → Haiku(**야간 Batches로 50% 할인**)가 갱신/안부 멘트 + 보험료 변화 요약 생성 → 가드레일 통과 → 인앱 **'오늘의 케어 N건' 할일 큐**.

**무노동 루프의 완성도**가 강점이다 — 설계사 검수 후 발송하면 **활동로그가 자동 기록(수기 입력 0)**되어 파이프라인 자동전이까지 이어진다. 신규는 트리거 추출 + 할일 큐 UI뿐이다. Premium 티어의 핵심 '영업 복리' 가치가 여기 있다.

### 발송 채널 현실화 — '원탭 자동발송' 거짓 전제 제거 (Phase2, 노력 high)

⚠️ **인프라 현실 직시 기능.** `Kakao.Share.sendDefault`는 수동 공유 시트라 자동/1:1 발송이 불가하고, 알림톡은 가변 AI 카피를 심사할 수 없으며, 친구톡은 수신동의·발송시간 규제를 받는다. 그래서 두 갈래로 정직하게 분리한다 — **개별 AI 멘트 = 클립보드 복사 + 카톡 앱 열기** / **대량 정형(만기·생일) = 사전심의 통과한 고정 알림톡 템플릿**. 후자는 `kakao_biztalk.py`(pending)를 풀어 수신거부·발송시간(08~21시)·건당 비용을 관리한다. 노력이 high인 이유는 알림톡 사전심의 + 스팸 거버넌스 + 정보통신망법 준수가 모두 묶여서다. (Open Question: 정형 알림톡 승인 가능 범위와 가변 AI 카피의 경계선.)

## 7. 소개 (Referral)

> 지인 소진 후 소개 전환 실패가 한국 설계사 도태의 핵심. 발굴 단계의 **바이럴 루프(유입)**와 소개 단계의 **리퍼럴 추적(전환)**은 같은 데이터 모델의 양면이다.

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| 리퍼럴 체인 추적 + 소개요청 타이밍 알림 | source/referred_by 추적 모델 부재, 소개 타이밍·멘트 감 | 규칙기반 타이밍(D+7) + 멘트는 메시지 엔진. 'AI 타이밍 감지'는 과욕이라 단순 규칙 | Phase2 | Customer(source/referred_by), 메시지 엔진, Notification | med / med |
| 고객 후기·만족도 1탭 수집 → 후기 카드 | 신뢰획득 메커니즘(후기 많은 설계사)인데 수집·디자인 역량 부재 | 4★+면 한줄평을 PII 제거·다듬어 인용 카드뉴스로 변환 (Sonnet) | Phase2 | promotion 카드 템플릿, SENS/카톡, Claude, Notification, Customer memo | med / low |

### 리퍼럴 체인 추적 — 자동귀속 데이터 위에만 얹는다 (Phase2)

소개 타이밍과 멘트를 감으로 잡던 문제를 푼다. **단, 'AI 타이밍 감지'는 과욕이라 단순 규칙으로 시작**한다 — 계약 직후/만기갱신 성공 직후 **D+7** 알림. 멘트만 메시지 엔진(목적=소개요청)으로 생성한다. 결정적 원칙: **반드시 바이럴 루프가 채운 `referred_by` 데이터 위에 얹어야** 하며 수기 입력에 의존하면 안 된다. 소개가 발생하면 `referred_by`로 체인을 트리 형태로 시각화한다. 신규는 referral 필드 + 체인 시각화 + 타이밍 규칙이다.

### 고객 후기·만족도 1탭 수집 → 후기 카드 — 평판 리스크 게이팅 (Phase2)

당근·블로그에서 '후기 많은 설계사'가 실제 신뢰획득 메커니즘인데 수집 채널·디자인 역량이 없었다. 온보딩 후 1탭 별점 설문 → **4★+ 면** Sonnet이 한줄평을 PII 제거·다듬어 인용 카드뉴스로 변환 + 소개요청 트리거. **핵심 게이팅**: 부정 후기(1~3★)는 외부 노출 0, 설계사에게 '케어 필요' 알림으로만 사용해 평판 리스크를 차단한다. promotion 카드 템플릿을 그대로 재활용한다.

## 8. 브랜딩 (Personal Branding)

> **후행 활성화 기둥.** 파이프라인 신뢰가 쌓인 뒤 얹는다. 광고심의 회피를 위해 **보장내용·수익률·타사비교·경품을 원천 배제**하고 자기소개·연차·신뢰 위주 카피만 자동생성한다.

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| 성과·셀프브랜딩 자동 카피 (광고심의 안전) | SNS/판촉물 제작역량 부족, 보장내용 넣으면 심의 위반. 4중 중복 → 1개 통합 | achievement_data KPI → 자기브랜딩 카피 생성, 템플릿 슬롯 자동 채움 (Haiku) | Phase3 | promotion 14종 + achievement_data + 카카오 Share + Claude + 가드레일 | low / low |
| 네이버/당근 로컬 발견 SEO 콘텐츠 발행 | 고객 탐색 1순위는 네이버·당근인데 인스타/카톡은 발견 채널 아님 | 심의 안전 보험 상식·교육 콘텐츠를 블로그 긴글/동네생활 답변형으로 생성 (Sonnet) | Later | promotion 템플릿, Claude, 가드레일, 후기 수집 | med / med |

### 성과·셀프브랜딩 자동 카피 — promotion 14종의 재배치 (Phase3)

4개 페르소나가 4중으로 중복 제안한 것을 1개로 통합했다. `achievement_data` KPI를 Haiku가 받아 **보장내용·수익률·타사비교·경품을 원천 배제**한 자기브랜딩 카피를 생성하고, **promotion 14종 템플릿 슬롯을 자동으로 채운다**. 트리거는 월 1회로 통일(주간은 식상하고 KPI 변화도 미미) → '이번 달 성과 카드 준비됨' → 자동 채움 미리보기 → 미세조정 → 공유.

**워터마크는 강제하지 않는다** — 설계사가 싫어해 이탈하기 때문. 진짜 바이럴은 워터마크가 아니라 **추천코드 링크**가 만든다(브랜딩이 발굴의 바이럴 루프와 연결되는 지점).

### 네이버/당근 로컬 발견 SEO 콘텐츠 — 진짜 발견 채널 (Later)

인스타·카톡은 '이미 아는 사람'용이라 발견 채널이 아니라는 통찰이 깔려 있다. 한국 고객의 설계사 탐색 1순위인 **네이버 검색·당근 동네생활**을 겨냥해, Sonnet이 심의 안전한 보험 상식·교육 콘텐츠(상품명·수익률 배제)를 블로그 긴글/동네생활 답변형으로 생성하고 검색 인덱싱 구조를 반영한다. 후기 카드와 연동된다.

## 9. 전 단계 공통 — 안전망·측정·과금 (퍼널을 가로지르는 기능)

> 특정 단계에 속하지 않고 **모든 AI 기능의 선결·기반**이 되는 기능들. 특히 컴플라이언스·PII는 **이게 없으면 다른 AI 기능 전부 출시 불가**다.

| 기능 | 해결하는 문제 | AI 역할 | Tier | 재활용 기존자산 | 임팩트/노력 |
|---|---|---|---|---|---|
| 보험 컴플라이언스 가드레일 (공유 검열 미들웨어) | 모든 AI 텍스트가 광고심의·비교광고·특별이익 규제 리스크 | 정규식 차단 + Haiku가 경계 모호표현만 warn(block 권한 없음) + 사람 최종승인 | MVP | community/content_filter.py 정규식 + profanity_words.json 패턴 | high / med |
| PII 동의·민감정보 국외이전 동의 동선 | 병력=민감정보를 Claude API(미국)로 전송, 국외이전 별도 동의 필요 | 없음 (법무/동의 인프라) | MVP | customer-agree 페이지, share_token | high / med |
| AI 비용 게이팅 + North Star 측정 대시보드 | AI 메시지·진단 비용 누수, '분석 건수' 외 측정 지표 없음 | 없음 (거버넌스). prompt caching·Batches로 비용 절감 추적 | Phase2 | membership/credit.py, analytics 앱, GA4 | high / med |

### 보험 컴플라이언스 가드레일 — 해자가 아니라 '위생' (MVP, 모든 AI 기능의 게이트)

⚠️ **이게 없으면 다른 AI 기능을 출시할 수 없다.** 모든 AI 생성 텍스트가 단일 게이트(`core/ai_guardrail.py`)를 통과한다. 3중 구조 — ①결정적 정규식 차단(금칙어·경품·보장금액 패턴, **block은 정규식만**) + ②Haiku가 경계 모호표현만 **warn(block 권한 없음, LLM verdict를 자동 신뢰 안 함)** + ③사람 최종승인. 기존 `community/content_filter.py` 정규식 구조와 `profanity_words.json` 패턴을 재활용한다.

**포지셔닝이 곧 법적 방어다.** '심의 안전 배지'를 붙이는 순간 Foliio가 보증책임을 떠안아 소송 사유가 된다. 그래서 **'1차 실수 방지 보조, 최종책임은 설계사'** 면책 UI로 한정한다. 비교광고('타사 대비 2배')·특별이익('사비로 첫 달 대납')·불완전판매 같은 **맥락 위반은 필터가 못 잡음을 명시**해야 한다. 출시 전 위반 카피 100건 골든 회귀 테스트는 필수. (Open Question: 면책 카피·동의 로그 구조의 법적 유효성은 보험업법 준법감시 전문가 자문 필요.)

### PII 동의·민감정보 국외이전 동의 동선 — 모든 AI 기능의 선결 과제 (MVP)

⚠️ **법무 검토 결과가 AI 기능 전체의 출시 가부를 결정한다.** 고객 병력=민감정보를 Claude API(Anthropic 미국)로 전송하므로, 개인정보보호법상 **민감정보 국외이전 별도 동의**가 필요하다. AI 기능 진입 전 `customer-agree`가 'AI 분석 위한 국외 제3자 제공'을 커버하는지 점검·확장한다. 구현은 동의서에 민감정보 국외이전 항목 추가 + 1탭 구두동의 로그 + `share_token` 만료/회수/조회로그/robots 차단이다. **바이럴 루프의 지인 B 셀프입력은 제3자 수집 소지가 있어 별도 동의가 필요**하다. (Open Question: 현 customer-agree로 커버 가능한가, 아니면 별도 동의서·국내 처리 대안(온프레미스/마스킹 후 전송)이 필요한가.)

### AI 비용 게이팅 + North Star 측정 대시보드 (Phase2)

멤버십 크레딧이 OCR/customer 생성에만 묶여 있어 AI 메시지·진단 비용이 샌다. AI 호출량을 `monthly_credit`에 차감 연결하고 비용 상한·알림을 건다(prompt caching·Batches로 절감 추적). **북극성 = '미팅당 share_token 발송수 × 고객 열람률 × 소개 귀속 리드수'** 이벤트를 설계하고 멘트 톤 A/B를 추적한다. (Open Question: 발송이 클립보드 복사 기반이면 발송 이벤트를 어떻게 정확히 계측하는가.)

## 부록 A. 미해결 질문 (PM 의사결정 대기)

이 기능 카탈로그를 실행으로 옮기기 전에 확정해야 할 항목:

1. **표준 보장 기준선의 권위** — 연령/성별/직업위험별 적정 보장 밴드의 출처(금감원/보험연구원/자체 컨센서스)와 면책 문구. → 진단 기능 출시 가부 결정.
2. **민감정보 국외이전 동의 범위** — 현 customer-agree로 커버 가능한가, 별도 동의서·국내 처리 대안(온프레미스/마스킹)이 필요한가. → AI 기능 전체 출시 가부 결정.
3. **알림톡 사전심의 승인 범위** — 만기/생일 정형 알림의 실제 승인 가능 범위, `kakao_biztalk.py` pending 해제 시점, 가변 AI 카피와 정형 알림톡의 경계선.
4. **지인 B 제3자 수집 동의 동선** — 1탭 동의 설계, `share_token` 만료/회수/조회로그/robots 차단 구현 수준.
5. **컴플라이언스 면책 구조의 법적 유효성** — '심의 안전 배지' 대신 어떤 면책 카피·동의 로그가 Foliio를 보호하는가. 보험업법 준법감시 전문가 자문.
6. **북극성 발송 이벤트 계측** — 클립보드 복사 기반 발송을 어떻게 정확히 측정하는가.
7. **무료 분석 → 유료 전환율 가설** — '분석=무료, 발송/자동화=유료' 매핑이 실제 지불의사로 이어지는지 베타 데이터로 검증.
8. **라이브 프레젠테이션 수요** — 상위 설계사의 '고객 앞 데모'가 실제 차별화·입소문 가치인지 사용자 인터뷰.
9. **신입 콜드스타트 aha** — 샘플 증권·일반 상식 콘텐츠가 첫 리텐션을 끌어올리는가, 진짜 고객 데이터가 있어야만 aha가 발생하는가.
