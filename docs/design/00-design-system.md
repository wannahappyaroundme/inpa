# 인파(Inpa) 디자인 시스템

> 컬러 · 타이포그래피 · 스페이싱 · 컴포넌트 · 데이터밀도 UI · 레퍼런스 픽 · 디자인 원칙
> 대상 제품: **인파(Inpa)** — 보험설계사가 고객 앞에서 데이터로 신뢰를 증명하는 영업지원 OS
> 모든 임계 수치는 **(추정)**, 법무 기준선(G4) 확정 전까지 토큰 상수 + `(추정)` 주석 강제.
> 메인 DNA: **무신사(절제·격자) × 직방(딥블루 신뢰·데이터 시각화)** — UI는 비키고 데이터(히트맵)가 주인공.

---

## 0. 이 문서의 한 줄 요약

인파는 **100+ 셀 히트맵 / 갈아타기 비교안내서**를 한 화면에 띄우는 데이터 밀도 제품이다. 따라서 디자인 시스템의 제1 책무는 "예쁘게"가 아니라 **색이 의미를 흐리지 않게** 하는 것이다. 핵심 결정 두 가지:

1. **인디고 역할분리** — 브랜드색 `#1E40C4`(헤더/CTA/로고)와 데이터의미색 `#3B5BDB`(충족셀/제안선)를 hex 차원에서 분리. 브랜드색은 셀에 못 쓰고, 데이터색은 헤더/CTA에 못 쓴다.
2. **정직성 레드라인 = 시스템 강제** — 안전배지/심의완료/AI보장 카피를 디자인 라이브러리에서 **슬롯 자체 물리삭제**. 색으로 위협을 과장하지 않게 amber(부족) ≠ red(위험)를 분리.

## 1. 컬러 시스템

### 1.1 컬러 토큰 전체 표 (hex + 용도)

#### Primary — 브랜드 (헤더/CTA/로고 전용)

| 토큰명 | hex | 용도 | 잠금 규칙 |
|---|---|---|---|
| **Inpa Indigo (Brand)** | `#1E40C4` | 브랜드 · 헤더 · 1차 CTA · 로고 심볼 | **셀/제안선에 사용 금지** |

> **왜 `#1E40C4`인가**: 히트맵 충족셀 + 비교표 제안선이 점유한 `#3B5BDB`보다 한 톤 깊은 딥인디고. 밀도 화면에서 "브랜드 강조 vs 충족 데이터 vs 제안선"을 색으로 구분 가능하게 만드는 분리선. BI안 `#2D4BC4`(구분 약함)와 PM안 `#1B3A6B`(CTA 활력 부족) 사이로 조정 — 명도차 충분 + 보험 신뢰 딥블루(직방 DNA) 유지. foliio 상속 teal `#00C5D1`은 **deprecate**.

#### Secondary — 제안 · 기존 · 포인트 · 앵커

| 토큰명 | hex | 용도 | 비고 |
|---|---|---|---|
| **Proposal Indigo** | `#3B5BDB` | 비교표 제안컬럼 · 제안선 · 히트맵 **충분셀** | **데이터 의미 전용잠금** (헤더/CTA 금지). foliio 상속 토큰 그대로 |
| **Existing Mint** | `#12B5A4` | 기존 보유 보험데이터 · 비교표 좌측 "기존" | 제안 인디고와 보색계 대비 → Before/After 좌우 분리 (강남언니 DNA) |
| **Point Orange** | `#F76707` | 발굴탭(엄지영역 하단중앙) 강조 · 보조 CTA | 직방식 단일 활력점 — **1화면 1회만** |
| **Deep Navy Anchor** | `#1B2A57` | 푸터 · 신뢰 앵커 · 앱아이콘 배경 | 인디고보다 무거운 딥블루 (명함 흑백/금박 대체) |

#### Semantic — 상태

| 역할 | hex | 용도 |
|---|---|---|
| success | `#2F9E44` | 충분 · 성사 · 동의완료 |
| warning | `#F59E0B` | 부족 (개선여지) — **위험 아님** |
| danger | `#E03131` | 위험 · 해지손해 · §97 불리점 · 부당승환 · 미동의 만료 |
| info | `#3B5BDB` | 정보 안내 (제안색과 동일 계열) |

#### 히트맵 3색 (데이터 전용)

| 상태 | hex | 셀 표현 | 패턴(이중 인코딩) |
|---|---|---|---|
| **충분 (enough)** | `#3B5BDB` | 연한 틴트 fill + 인디고 텍스트/보더 | 채움 |

| gray-600 | `#6B7280` | 본문 · 라벨 |

| gray-300 | `#CED4DA` | 보더 · 구분선 |
| gray-200 | `#E5E8EB` | 카드 보더 · 분할선 |
| gray-050 | `#F7F8FA` | 배경 · 미분류 셀 surface |

### 1.2 인디고 충돌 해결 규칙 (명문화)

```
┌──────────────────────────────────────────────────────────────┐
│  역할분리로 인디고 충돌 = 0                                      │
├──────────────────────────────────────────────────────────────┤
│  브랜드  #1E40C4  →  헤더 · CTA · 로고      ✗ 셀에 못 씀        │
│  데이터  #3B5BDB  →  충족셀 · 제안선        ✗ 헤더/CTA에 못 씀  │
└──────────────────────────────────────────────────────────────┘
```

- **왜 필요한가**: 밀도 화면에는 브랜드 강조(헤더 로고/CTA 버튼) + 충족 데이터(히트맵 셀) + 제안선(비교표)이 **동시 노출**된다. 셋이 같은 인디고면 사용자는 "이 파란 건 브랜드인가, 충족 데이터인가, 제안인가"를 구분 못 한다.
- **빌드스펙 근거**: `07-product-build-spec.md` L415(히트맵 충분셀)·L590(비교표 제안선)이 이미 `#3B5BDB`로 못 박힘. 브랜드까지 같으면 구분 불가.
- **stylelint 강제**: 화면 코드는 semantic 레이어만 참조 → `#1E40C4`/`#3B5BDB` raw hex 직접 호출 금지 (§1.3).

### 1.3 토큰 3레이어 아키텍처 (SSOT)

```
primitive  →  semantic  →  component
(원자색)      (역할, 화면이      (히트맵 셀이
              참조하는 유일       직접 소비)
              레이어)
```

```scss
// ── layer 1: Primitive (의미 없음) — _tokens-primitive.scss ──
$indigo-brand: #1E40C4;
$indigo-data:  #3B5BDB;
$mint-500:     #12B5A4;
$orange-500:   #F76707;
$navy-900:     #1B2A57;
$amber-500:    #F59E0B;
$red-500:      #E03131;
$green-500:    #2F9E44;
$gray-400:     #ADB5BD;
$ink-900:      #16181D;

// ── layer 2: Semantic (화면이 참조하는 유일 레이어) — _tokens-semantic.scss ──
$brand-primary:    $indigo-brand;  // 헤더/CTA/로고 — 셀 금지
$color-proposal:   $indigo-data;   // 제안선/충분셀 — 헤더 금지
$color-existing:   $mint-500;      // 기존 보유
$color-point:      $orange-500;    // 발굴탭 단일 활력
$color-anchor:     $navy-900;      // 푸터/앱아이콘
$color-success:    $green-500;
$color-warning:    $amber-500;     // 부족 ≠ 위험
$color-danger:     $red-500;       // 위험/§97 불리점
$color-gap:        $gray-400;      // 공백/미분류

// ── layer 3: Component (히트맵 셀) — _tokens-component.scss ──
$heatmap-cell-enough:       $color-proposal;
$heatmap-cell-short:        $color-warning;
$heatmap-cell-none:         $color-gap;
$heatmap-cell-unclassified: transparent; // 점선 + $color-gap border
```

**SCSS → CSS 변수 브리지 (필수)** — 히트맵 100+ 셀은 런타임 색칠인데 SCSS는 컴파일타임이라 분기 불가. semantic을 `:root`에 미러링:

```scss
:root {
  --brand-primary:   #{$brand-primary};
  --proposal-indigo: #{$color-proposal};
  --short-amber:     #{$color-warning};
  --gap-gray:        #{$color-gap};
}
```
→ TS는 `statusOf(rate) → 'var(--proposal-indigo)'` 한 줄로 매핑.

**강제 규칙 체크리스트**
- [ ] 화면 코드는 **layer 2만** 참조 (raw hex / layer1 직접 호출 금지)
- [ ] stylelint `color-no-hex` 활성 — 위반 시 CI 빨강
- [ ] 히트맵 셀은 CSS 변수 클래스(`.cell--enough`)로 색칠 — `[ngStyle]` 인라인 hex 금지 (리플로우·CSP)
- [ ] foliio 잔존 하드코딩(`#00C5D1` / `#FF60B6`) `grep -rn` 제거 → 토큰 외 0건
- [ ] `$global-header-h` 단일출처를 토큰 레이어로 흡수 (3곳 동기화 트랩 제거)

### 1.4 임계치 토큰화 (추정)

```ts
// coverage-thresholds.ts — 단일 상수 (SCSS와 1:1)
export const THRESHOLD_SUFFICIENT = 0.8; // (추정) 충족 80% — 법무 G4 확정 전
export const THRESHOLD_GAP        = 0.3; // (추정) 공백 30% 미만

export function statusOf(rate: number): 'enough' | 'short' | 'none' {
  if (rate >= THRESHOLD_SUFFICIENT) return 'enough';
  if (rate >= THRESHOLD_GAP)        return 'short';
  return 'none';
}
```
- 셀 색을 결정하는 **유일한 함수**는 `statusOf(rate)`. 정책 변경 시 상수 1곳만 수정.
- UI에는 `<inpa-estimate-badge>` 컴포넌트 1개로 "(추정)" 표시 — 법무 기준선 확정 시 제거.

## 4. 핵심 컴포넌트

### 4.1 카드 (action-card)

```
┌────────────────────────────────────┐
│▎김설계 고객  만기 D-14   [연락하기] │  ← 좌측 4px 액센트바 = 우선순위
└────────────────────────────────────┘
```
- 1줄 = 고객명 + 할일 + [액션]. 둥근사각 카드피드 (야놀자 패턴만 차용, 칸반 아님).
- 좌측 4px 액센트바 = 우선순위 (위험=amber, 일반=gray). **자동 이벤트로만 채워짐 (수기 입력 0)**.
- boder-1 + shadow-1 단계만. 색 = 무채색 70 + 인디고 20 + 시맨틱 10.

### 4.2 테이블 (compare-ledger)

```
┌─────────────┬──────────────┬──────────────┐
│   담보       │ 기존(민트)    │ 제안(인디고)  │
├─────────────┼──────────────┼──────────────┤
│ 암진단비     │  3,000만원    │  5,000만원    │
│ 차액         │           +2,000만원 (굵게)   │
│ ▎해지손해    │  환급손해 발생가능 (적색 배지) │  ← 불리점 = danger
└─────────────┴──────────────┴──────────────┘
   §97 6항목 체크 ☐☐☐☐☐☐  →  [PDF] 회색 비활성 (서명 전 잠금)
```
- 좌(기존 민트) · 우(제안 인디고) 2열 원장형. 차액 행 굵게. tabular-nums 자리정렬.
- 불리점 행 = `danger #E03131` 적색 경고 배지 (amber 아님 — 위험).
- §97 6항목 체크 전 [PDF] 버튼 **회색 비활성** (상태머신). 판정은 BE 룰엔진 산출 → FE는 표시만.

### 4.3 히트맵 셀 (heatmap-cell)

| 상태 | fill | 보더 | 텍스트 | 패턴 |
|---|---|---|---|---|
| 충분 | 연한 인디고 틴트 | — | 인디고 `#3B5BDB` | 채움 |
| 부족 | 연한 amber 틴트 | 좌측 4px `#F59E0B` | ink | 좌측 액센트바 |
| 없음 | 투명 | 점선 `#ADB5BD` | gray | 점선 |
| 미분류 | gray-050 | 점선 회색 | "미분류" | 점선 칩 |

- 셀 크기: 모바일 **24×24** ~ 데스크톱 **32×32**, 간격 2px.
- **fill은 연한 틴트, 텍스트/보더만 진하게** — 한 화면에 기존민트+제안인디고+충족3색+위험적색 4종 동시 노출 시 진한 fill이면 가독성 붕괴.
- 색 + 패턴 **이중 인코딩** (WCAG 1.4.1) — 색만으로 정보전달 금지.

## 8. 레퍼런스 분석표 + 인파 픽 & 차용 전략

### 8.1 5 레퍼런스 분석표

| 레퍼런스 | DNA | 인파가 **취하는** 것 | 인파가 **버리는** 것 | 적용 영역 |
|---|---|---|---|---|
| **직방** (1픽) | 데이터 시각화 + 딥블루 신뢰 | 딥블루 신뢰톤 + 데이터 정교함 + 발굴 포인트 오렌지 | 3D/지도급 화려효과 → 표 밀도만 | 키컬러 · 히트맵 · 비교표 전문성 |
| **무신사** | 무채색 절제 격자 | 무채색 70% / 1px 보더 / 그림자 최소 / 격자 → 데이터가 주인공 | 패션 블랙 강조 → 인디고 신뢰톤 치환 | 전체 밀도 철학 · 색 절제 |
| **강남언니** | 불투명 도메인 투명화 | Before/After 좌우대칭 (기존 민트 / 제안 인디고) | 친근/캐주얼 톤 → 절제 (레드라인 충돌) | 갈아타기 비교안내서 |
| **리디** | 텍스트 위계 | tabular-nums 숫자 위계 / 행간 여백 텍스트 계층 | — | 타이포 위계 |
| **야놀자** | 액션큐 카드 | 둥근사각 카드피드 패턴**만** | 핑크/경쾌 모션/칸반 (신뢰 도메인 부적합) | 액션큐 홈 |

> **마스코트 / 바운스 모션 전면 폐기.**

### 8.2 메인 DNA = 무신사 × 직방 (합성)

```
무신사 = 절제된 그릇   →  100+ 셀 히트맵에서 UI가 비키고 데이터가 주인공
직방   = 신뢰의 내용물  →  그 위에 딥블루 신뢰톤 + 데이터 정교함을 얹음
```
- 두 픽은 **대립이 아니라 층위**다 (무신사=절제 그릇 / 직방=신뢰 내용물).
- **단일 1픽 강제 시 → 직방.** 데이터 시각화로 전문성을 증명하는 딥블루 신뢰톤이 인파 사업 본질(히트맵/비교안내서로 설계사를 고객 앞에서 전문가로 보이게)과 정확히 일치, 보험 민감 도메인 신뢰 요구에 직결.

### 8.3 차용 전략 요약

- **취한다**: 직방 딥블루+데이터 밀도 / 무신사 무채색 절제·격자 / 강남언니 Before/After / 리디 숫자 위계 / 야놀자 카드 패턴.
- **버린다**: 직방 3D 화려효과 / 무신사 패션 블랙 / 강남언니 캐주얼 톤 / 야놀자 핑크·경쾌 모션·칸반 / 마스코트·바운스 전면.
