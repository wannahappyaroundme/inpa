# 27. 보험증권 추출 검토 운영·비공개 평가 계약

> 목적: 보험증권 자동 정리의 운영 수치와 비공개 평가 결과를 개인정보 없이 해석하고, 어떤 경우에 추출 기능을 열지 않을지 고정한다. 이 문서의 평가는 보험증권 **추출 단계만** 다룬다. 동의, 분석 합산, 공유 스냅샷, 계정 격리, 동시 처리의 출시 기준은 각각의 통합 테스트와 부하 검증이 별도 권위다.

## 1. 운영 수치의 권위

- Claude 호출 수, 토큰, 추정 비용은 `ClaudeApiLog` 한 곳만 합산한다. 추출 결과에 비용 사본이 있어도 더하지 않는다.
- 작업 상태, 대기·처리·검토 시간, 후보·확정 수, 수정 행동 수는 보험 추출 작업과 결과에서 계산한다.
- 비용은 실제 청구액이 아니라 토큰과 모델 단가로 계산한 추정값이다. 원천 수치는 토큰이다.
- 원문, 상품명, 담보명, 파일명, 저장 경로, 고객·설계사 정보는 운영 응답과 로그에 포함하지 않는다.
- 동시 실행 상한을 관리자 화면에서 바꾸면 이미 가져간 작업은 기존 값으로 마치고, 다음 작업 가져오기부터 새 값이 적용된다.
- 기능 스위치와 원본 보존시간은 배포 환경의 읽기 전용 값이다. 관리자 API에서 바꾸지 않는다.

## 2. 수치 읽는 법

모든 정확도 비율은 `맞은 수 / 비교 가능한 전체 수`와 비율을 함께 표시한다. 비교 가능한 값이 0개면 비율은 0이 아니라 `null`이다.

| 수치 | 분자 / 분모 |
| --- | --- |
| 보험사·상품명·계약일·만기일·월 보험료 일치 | 값이 정확히 같은 증권 수 / 전체 증권 수 |
| 담보 재현율 | 연결된 정답 담보 수 / 전체 정답 담보 수 |
| 담보 정밀도 | 연결된 추출 담보 수 / 전체 추출 담보 수 |
| 가입금액·담보보험료 일치 | 원 단위가 정확히 같은 연결 담보 수 / 연결 담보 수 |
| 표준 담보 위치 일치 | 3단 위치가 같은 연결 담보 수 / 정답 위치가 있는 연결 담보 수 |
| 검토 포착률 | 확인 항목으로 보낸 실제 오류 수 / 전체 실제 오류 수 |
| 잘못 보낸 확인 항목 비율 | 확인 항목으로 보낸 정확한 값 수 / 전체 정확한 값 수 |
| 공급자 실패율 | 공급자 실행 실패 증권 수 / 전체 증권 수 |

- 원 단위 금액은 1원 차이도 오류다. 정답이 비어 있는데 값이 있거나 그 반대인 경우도 중대한 금액 오류로 센다.
- 지연시간 p50·p95는 정렬한 값에서 `ceil(p × 표본 수) - 1` 위치를 고르는 nearest-rank 방식이다. 표본 수를 항상 함께 본다.
- 공급자 호출시간과 PDF 처리부터 검증까지의 전체 시간을 분리한다.
- 감지 후보 수와 최종 확정 수는 사실 수치이며 정확도나 성공률로 부르지 않는다.
- 수정률은 현재 작업 단위로만 해석한다. 같은 행을 여러 번 수정한 행동 수를 담보 수정률로 나누지 않는다.
- 검토 포착은 오류 단위가 `needs_review`, `no_evidence`, `unmatched`, `invalid`, `manual` 상태이거나 해당 오류 필드가 명시적으로 확인 항목에 들어간 경우만 센다. 단순히 화면에 보이는 `review_ready`는 포착으로 세지 않는다. 포착 수는 실제 오류 수를 넘을 수 없다.

## 3. 비공개 평가 자료 계약

평가 자료는 모든 git 작업공간 밖의 접근 통제 저장소에 둔다. PDF, 정답, 검토 결과, 개별 예측, 실패 원문은 git, CI 캐시·결과물, stdout·stderr, Sentry에 넣지 않는다.

최소 구성은 다음과 같다.

- 서로 다른 비식별 증권 100건 이상
- 사람이 확정한 담보 정답 1,000행 이상
- 생명·손해보험 각각 20건 이상
- 구형·현재 양식 각각 20건 이상
- 짧은·긴 증권 각각 10건 이상
- 보험사 코드 `1, 2, 7, 11, 12, 201, 206, 213` 각각 5건 이상

manifest는 이 기준을 높일 수는 있지만 낮출 수 없다. 모든 사례 ID와 PDF·정답·검토 결과 경로는 고유해야 한다. 상대경로의 일반 파일만 허용하며 `..`, 절대경로, 저장소 밖으로 향하는 심볼릭 링크는 실행 전에 거절한다. dataset root나 manifest가 git 작업공간 안에 있거나 작업공간을 부모로 포함하는 경우도 거절한다. 각 PDF·정답·검토 결과의 심볼릭 링크를 해석한 실제 경로도 모든 git 작업공간과 다시 대조한다.

스키마는 허용 필드만 받는다. 고객명, 계약자·피보험자, 전화, 이메일, 주소, 주민·증권·계좌번호 같은 식별 필드가 한 개라도 추가되면 평가하지 않는다. 비식별 상품명과 담보명도 채점 메모리 안에서만 쓰고 집계 출력에는 내보내지 않는다.

## 4. 정답과 검토 결과의 분리

- 독립 판정자가 `truth`를 만든다.
- 정답을 보지 않은 검토자가 실제 검토 화면으로 수정·확정한 결과를 별도 `reviewed output`으로 만든다.
- 두 파일은 같은 파일이거나 같은 내용을 복사한 결과여서는 안 된다.
- 검토 결과 봉투에는 허용 형식의 불투명한 `reviewer_ref`, UTC 검토 시각, 검토 도구 버전, 결과 스키마 버전, `truth_access=false`, 예측 내용 해시를 기록한다. 이 값들은 파일 작성자가 넣는 자기선언이며 신뢰 증명이 아니다. 내용 해시는 파일 무결성과 단순 복사 실수를 확인할 뿐, 독립 검토를 증명하지 않는다.
- 검토 결과가 준비되지 않았으면 `review/post_review = not_measured`로 표시한다. 정답을 복사해 검토 후 100%를 만들지 않는다.
- live adapter에는 PDF와 비식별 실행 계약만 전달한다. 정답·검토 결과·그 경로는 전달하지 않는다.

현재는 신뢰 가능한 서버 측 서명 감사 또는 검토 화면 export 출처 검증기가 없다. 따라서 구조가 유효한 검토 결과도 항상 `review/post_review = not_verified`, `TRUSTED_REVIEW_AUDIT_UNAVAILABLE`로 보고하며 수치는 참고용으로만 표시한다. `reviewer_ref`, `truth_access`, 내용 해시가 모두 정상이어도 출시 권한은 생기지 않는다. 향후 신뢰 검증기는 별도 보안 설계와 승인을 거쳐야 하며, 이 평가기가 서명이나 신뢰 값을 대신 만들지 않는다.

## 5. 담보 행 연결 규칙

1. 정답과 예측의 `source_ref(page, line_start, line_end)`가 같은 행을 먼저 연결한다. 정답 source ref는 문서 안에서 고유해야 한다.
2. 줄 근거가 없는 기존 방식만 Unicode NFC와 양끝 공백을 정리한 담보명을 사용한다. 같은 이름은 문서 출현 순서의 다중집합으로 연결한다.
3. 중복 이름의 개수가 달라 어느 행인지 확정할 수 없으면 모호한 행으로 보고한다. 금액·표준 위치가 가장 비슷한 행을 임의로 고르지 않는다.
4. 모호한 행은 금액·표준 위치 일치율 분모에서 제외하며 개수를 별도로 표시한다.
5. 금액이 틀려도 이미 정해진 행 연결은 바꾸지 않는다. 그래야 금액 오류가 누락과 가짜 행으로 바뀌지 않는다.

`review/pre_review`와 `review/post_review`는 담보명 연결을 사용하지 않는다. 모든 확정 담보 행에 고유한 `source_ref`가 있어야 한다. 근거가 없거나 같은 근거가 중복되면 재현율·금액·표준 위치 점수를 올리지 않으며, 출시 기준의 근거 연결 계약도 실패한다.

조용한 누락은 정답 담보가 예측 행, 로컬 후보, 미매칭 또는 확인 항목 어디에도 나타나지 않은 경우다. 로컬 후보로 드러난 누락은 재현율 오류이지만 조용한 누락은 아니다.

## 6. 실행 순서와 데이터 수명

평가 실행은 접근 권한이 분리된 staging shell에서만 수행하며 실제 공급자 실행에는 별도 승인이 필요하다.

1. CLI 값과 split을 확인한다.
2. dataset root와 manifest가 모든 git 작업공간 밖인지 확인한다.
3. 전체 manifest, 경로, 정답, 검토 결과 스키마를 읽고 중복을 확인한다.
4. 100건·1,000행과 모든 층별 최소 수를 확인한다.
5. 위 검증을 모두 통과한 뒤에만 adapter를 만들고 PDF를 읽는다.
6. legacy와 review 각각에 대해 실제 사용한 모델, 결과 형식, 프롬프트, 정규화 스냅샷을 별도 실행 계약으로 기록하고 기대 계약과 비교한다. 두 파이프라인의 프롬프트·결과 형식·정규화가 다르면 서로 같다고 표시하지 않는다.
7. 예측은 메모리에서 집계한 뒤 버린다. 작업·보험·비용 원장에 저장하지 않는다.
8. stdout에는 전체 건수, 분자·분모·비율, 지연, 토큰·비용, 안전한 오류 코드와 예외 타입별 건수만 남긴다.

상세 출력을 여는 옵션은 제공하지 않는다. 실패 예외 메시지, 로컬 경로, 원문 또는 공급자 응답은 로그로 남기지 않는다. 기존 방식 실행 중에는 legacy logger를 평가 adapter 경계에서 잠시 차단하고 실행 후 원상 복구한다.

## 7. 비교 행과 현재 권한 한계

보고서는 기본 비교의 출시 권위 세 행과 별도 진단 행을 구분한다.

- `legacy/pre_review`: 기존 방식 기준선. 수치는 보고하지만 출시 차단에 쓰지 않는다.
- `review/pre_review`: 줄 근거, 새 추출, 코드 검증까지의 사람 수정 전 결과.
- `review/post_review`: 검토자가 수정·확정했다고 자기선언한 결과. 현재는 신뢰 출처 미검증 상태다.
- `openai_review/pre_review`: 같은 비식별 입력을 이용한 별도 공급자 진단 행. 출시 차단 판단의 권위는 없다.

출시 차단 확인을 요청하면 다음 계약을 검사한다.

- 출시 권위인 `legacy/pre_review`, `review/pre_review`, `review/post_review` 세 행이 모두 있어야 한다. 일부 행만 고르는 실행이나 `openai_review/pre_review`를 추가한 실행은 진단 결과를 함께 낼 수 있지만, 별도 진단 행이 출시 권위를 대신하지 않는다.
- `legacy/pre_review`: 해당 기존 방식 실행 계약 일치
- `review/pre_review`: 조용한 누락 0, 고유 근거 연결 계약 일치, 해당 새 방식 실행 계약 일치
- `review/post_review`: 신뢰 가능한 서버 측 감사 검증 완료가 먼저 필요하다. 현재 모든 파일은 `not_verified`이므로 이 조건을 통과하지 못한다.
- 어느 비교 경로든 공급자 실행 실패가 있으면 완전한 비교가 아니므로 실패
- 검토 후 결과가 없는데 출시 기준 확인을 요청하면 실패

`review/pre_review`를 포함한 일부 비교는 출시 기준 확인 옵션 없이 진단용으로 실행할 수 있다. 이 경우 종료 코드 `0`은 평가 실행 성공만 뜻하며 출시 승인이나 기능 스위치 개방 근거가 아니다. 현재 신뢰 검증기가 없으므로 이 명령만으로 추출 출시 승인을 만들 수 없다.

## 8. 종료 코드

| 코드 | 의미 | 다음 행동 |
| --- | --- | --- |
| `0` | 출시 기준 확인을 요청하지 않은 진단 평가가 정상 종료 | 집계 수치만 검토하며 출시 승인으로 사용하지 않음 |
| `1` | 출시 차단 확인 요청, 검토 후 결과 미측정·미검증, 공급자 실행 실패 중 하나 | 기능 스위치를 열지 않고 신뢰 감사 검증과 실패 유형을 먼저 해결 |
| `2` | 공급자 호출 전 CLI, 스키마, 경로, 중복, 개수, 층화 계약 실패 | private 자료 계약부터 수정 |

## 9. 운영 안전 점검

- 실제 holdout 실행 전 실행자와 저장소 접근 권한을 확인한다.
- 실행 중 네트워크와 공급자 사용 승인이 없으면 합성 가짜 adapter 테스트까지만 수행한다.
- 결과를 공유할 때는 집계표만 옮기고 사례 ID, 파일명, 원문, 상품·담보명을 옮기지 않는다.
- `0`건 오류는 100% 정확성을 통계적으로 증명하지 않는다. 보험사·양식별 표본과 새 실패 유형을 계속 추가한다.
- 기능 스위치는 향후 신뢰 가능한 검토 출처 검증, 이 평가, 통합 검증, PM 승인이 모두 끝난 뒤 배포 환경에서만 연다.

## 10. OpenAI 비공개 비교 평가

OpenAI 비교는 운영 대체 공급자나 이중 분석이 아니다. 운영 추출은 Claude 한 경로를 유지하고, `openai_review/pre_review`는 접근 통제된 staging shell에서 같은 비식별 입력과 같은 구조화 결과 스키마를 비교하는 진단 행으로만 쓴다. 이 행은 출시 차단 판단, 검토 후 확정값, 운영 실패 시 자동 전환의 권한이 없다.

실행 환경에는 운영 의존성과 분리된 `requirements-eval.txt`를 설치하고 `OPENAI_EVAL_API_KEY`, `OPENAI_EVAL_MODEL`을 둘 다 주입한다. 어느 하나라도 비어 있으면 PDF 처리 전에, 평가용 SDK가 없으면 공급자 호출 전에 안전 코드로 종료한다. 모델 ID 기본값은 없으며 공급자 응답은 저장하지 않는다(`store=False`). 요청에는 운영과 같은 로컬 마스킹 결과만 포함하고, 응답은 같은 구조화 스키마와 개인정보 출력 차단을 통과한 뒤 메모리에서만 집계한다. 공급자와 HTTP 클라이언트의 요청 디버그 로그는 각 호출 동안 차단하고 성공·실패 뒤 원래 로깅 상태로 복원한다.

```bash
pip install -r requirements-eval.txt
python manage.py eval_insurance_extraction \
  --dataset-root /private/eval-data \
  --manifest /private/eval-data/holdout.json \
  --split holdout \
  --compare legacy,review,openai_review
```

명령의 출력은 `openai_review/pre_review` 집계만 추가한다. 사례 ID, 파일명, 원문, 개별 예측은 출력하지 않는다. 이 비교는 비식별 텍스트에서 구조화 추출 차이를 측정할 뿐이며, 스캔 이미지 인식 정확도나 전체 보험 분석 정확도를 증명하지 않는다. 실제 정답 자료와 실행 승인이 없으면 합성 테스트까지만 수행한다.

선택한 `openai_review` 실행에서 공급자 실패가 발생하면 불완전한 비교 실행이므로 종료 코드 `1`을 반환한다. 이는 진단 실행의 완결성 규칙이며 OpenAI 행에 출시 권위를 주는 뜻이 아니다.

OpenAI 단가는 현재 인파의 승인된 비용표에 없으므로 이 행의 추정 비용은 `0`으로 두고 입력·출력 토큰만 권위 값으로 본다. Claude 보수 단가를 OpenAI 비용처럼 표시하지 않으며, 비용 비교가 필요하면 별도 가격표 검토와 승인을 먼저 거친다.

## 11. 합성 staging 동시 처리 점검

`render.staging.yaml`은 운영과 공유하지 않는 전용 자원만 연결한다. PostgreSQL 서비스 이름은 `inpa-insurance-db-staging`, 실제 데이터베이스 이름은 `inpa_insurance_staging`, 플랜은 `basic-256mb`, Key Value 이름은 `inpa-insurance-queue-staging`이다. 비공개 저장 버킷은 외부에서 `inpa-insurance-staging`으로 따로 만들고 staging 전용 접근 키와 endpoint만 넣는다. web, worker, cleanup은 모두 관리형 staging PostgreSQL에 직접 연결한다. 새 Blueprint의 검토, 비교, 병력 분석, 합성 부하 스위치는 각각 `False`로 고정해 시작한다. 운영 `render.yaml`에는 합성 부하 스위치를 넣지 않으며 코드 기본값 `False`를 유지한다.

이 Blueprint 동기화는 유료 Render PostgreSQL·web·worker·Key Value·cron 자원을 만들 수 있다. PM의 배포·비용 승인을 따로 받기 전에는 동기화하거나 생성하지 않는다. 승인 후에도 먼저 R2 버킷과 staging 전용 키를 만들고, Blueprint 값과 실제 자원 이름이 정확히 일치하는지 검토한다.

외부 실행 승인을 받은 staging shell에서만 `INSURANCE_REVIEW_GATE_ENABLED=True`와 `INSURANCE_LOAD_TEST_ENABLED=True`를 잠시 함께 설정한다. 준비·정리 명령은 파일이나 DB를 바꾸기 전에 PostgreSQL 실제 이름이 `inpa_insurance_staging`인지, 버킷 이름이 `inpa-insurance-staging`인지, `insurance_sources`가 그 버킷을 쓰는 S3 저장소인지, 입력 호스트에 `staging`이 들어가는지를 모두 확인한다. 하나라도 다르면 파일과 DB를 바꾸지 않는다. 실제 증권이나 `samples/`는 사용하지 않는다. 비공개 폴더는 모든 git 작업공간 밖에 만들고 폴더 `0700`, 파일 `0600`을 유지한다.

```bash
python manage.py prepare_insurance_load_fixture \
  --private-root /private/inpa-load/run-20260717-a \
  --run-id run-20260717-a \
  --expected-host inpa-be-staging.onrender.com

python scripts/load/insurance_import_concurrency.py \
  --base-url https://inpa-be-staging.onrender.com/api/v1 \
  --scenario /private/inpa-load/run-20260717-a/scenario.json \
  --auth-file /private/inpa-load/run-20260717-a/auth.json \
  --result /private/inpa-load/run-20260717-a/result.json \
  --workers 60 \
  --execute-staging run-20260717-a \
  --max-intake-p95-ms 5000 \
  --max-owner-wait-p95-ms 900000
```

준비 명령은 비밀번호 로그인을 쓸 수 없는 합성 설계사 20명, 각 고객 1명, 텍스트 층이 있는 합성 PDF 60개, 검토 확정 경쟁용 작업 4개와 token auth를 멱등 생성한다. 동일 내용 PDF를 서로 다른 설계사에게 함께 보내 해시가 같아도 계정별로 분리되는지 검사한다. 결과에는 전체 요청 수, 오류 수, 전체 지연과 설계사별 대기·완료 p95의 집계만 저장하며 설계사별 수치, 고객 ID, 작업 ID, token, 파일 경로는 저장하지 않는다.

각 상태 조회 HTTP 요청 제한은 최대 45초이고 전체 60건이 worker를 빠져나올 때까지 기다리는 시간은 별도 600초다. `max_owner_wait_p95_ms`는 접수 응답 시간이 아니라 설계사별 세 건의 생성부터 검토 필요 도달까지 p95를 다시 집계한 값에 적용한다.

실행 후 결과 파일을 접근 통제 위치로 옮긴 다음 그 파일을 비공개 폴더에서 제거하고 아래 명령을 실행한다. 정리 명령은 실행 이름과 합성 표식, 정확한 계정 범위가 맞고 폴더 안에 허용된 합성 파일만 있을 때 삭제한다. 중간 실패로 파일 일부만 생긴 상태도 안전하게 정리하며, 같은 정리를 두 번 실행하면 두 번째는 삭제 대상 0건으로 끝난다. 완료 직후 두 스위치를 다시 `False`로 돌린다.

```bash
python manage.py cleanup_insurance_load_fixture \
  --private-root /private/inpa-load/run-20260717-a \
  --run-id run-20260717-a \
  --expected-host inpa-be-staging.onrender.com
```

현재 외부 staging의 Render, 별도 PostgreSQL·Valkey·R2, Claude 환경값과 배포 인증이 준비되지 않은 환경에서는 외부 60건 실행을 통과했다고 기록하지 않는다. 로컬 PostgreSQL 16·Valkey 8·S3 호환 저장소 검증은 외부 staging 실행을 대신하지 않는다.
