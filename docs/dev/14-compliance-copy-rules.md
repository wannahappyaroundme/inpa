## 0. 한 줄 원칙

**인파는 보험을 중개·권유하지 않는다.** 고객에게 노출되는 모든 문구는 **'사실(fact)'만** 말하고, **'판정(judgment)'과 '권유(solicitation)'는 단 한 글자도 새지 않는다.** 판정·권유는 라이선스를 가진 설계사의 행위다. 이 문서는 그 경계를 **코드가 강제**하도록 만든다.

룰의 통제 지점은 셋이다 — **카피를 쓰는 사람(작성 가이드)**, **타입을 정의하는 사람(컴파일 차단)**, **응답을 내보내는 서버(런타임 차단)**. 세 겹 모두 통과해야 고객 화면에 글자가 닿는다.

## 1. 적용 범위 — 어디에 이 룰이 걸리나

문구를 **노출면(surface)** 으로 분류한다. 룰의 강도는 노출면에 종속된다.

| 노출면 | 코드 | 누가 보나 | 룰 강도 | 판정어 |
|---|---|---|---|---|
| 공유뷰A | `customer_facing` | 고객(라이선스 없음) | **최고(컴파일+런타임 양단 차단)** | 전면 금지 |
| 비교안내서 §97 출력 | `customer_facing` | 고객 | **최고** + 면책 상시 + §97 6항목 게이트 | 사실·차액만, 우열단정 금지 |
| AI 카톡 메시지(M6) | `customer_facing` | 고객 | **최고** + M5 가드레일 후처리 | 전면 금지 |
| 히트맵 | `planner_internal` | 설계사(라이선스) | 중(red 금지·neutral 디폴트) | 허용(설계사 기준 대비) |
| 대시보드·고객·기준설정 | `planner_internal` | 설계사 | 중 | 허용 |
| 랜딩·마케팅 | `marketing` | 불특정 | 중(안전배지·확정 금지) | 제품설명 한정 |

**핵심 분기**: `customer_facing`은 **블랙리스트 100% 차단 + 화이트리스트만 발화**. `planner_internal`은 판정어를 쓰되 **고객 공유로 새는 경로엔 자동 필터**(§6.4). 노출면 태깅이 룰 엔진의 1차 입력이다.

> (추정) 노출면 enum은 응답 직렬화 계층에서 컴포넌트/시리얼라이저 단위로 1개씩 명시 부여. 디폴트는 가장 엄격한 `customer_facing` (누락 시 안전측 실패).

## 2. 블랙리스트 — 고객 노출면 절대 금지어

### 2.1 금지 카테고리 5종

고객에게 새면 무자격 모집·부당권유·허위과장으로 직결되는 표현을 5개 의미군으로 묶는다.

| # | 카테고리 | 왜 금지(준법 근거) | 대표 토큰 |
|---|---|---|---|
| B1 | **적정성 판정** | 비라이선스 SW의 보장 적정성 판정 = 무자격 모집 | 부족 / 충분 / 넉넉 / 적정 / 모자라 / 빈틈 / 공백(판정맥락) |
| B2 | **갈아타기 권유** | 부당승환 권유(§97 위반) | 갈아타 / 갈아타세요 / 리모델링 / 재설계(권유맥락) / 바꾸세요 / 해지하고 |
| B3 | **안전·심의 사칭** | 인파가 심의·보증한다는 오인 | 안전 / 안심 / 심의완료 / 보장확정 / 검증완료 / 인증 / 적합판정 |
| B4 | **확정·단정** | 미래·효과 단정(허위과장 소지) | 무조건 / 100% / 반드시 / 확실히 / 절대 / 보장됩니다(이중의미) |
| B5 | **추천·우열** | 상품 추천·우열 단정 = 모집행위 | 추천 / 강추 / 최고 / 베스트 / 1위 / 유리합니다(단정) / 이게 낫 |

### 2.2 토큰 사전 구조 (JSON)

블랙리스트는 코드·검수·CI가 공유하는 **단일 출처(SSOT)** 다.

```jsonc
// inpa_fe/src/compliance/blacklist.tokens.json  (BE도 동일 파일 vendoring)
{
  "version": "2026-06-19",
  "surfaces_blocked": ["customer_facing"],
  "categories": {
    "B1_adequacy": {
      "label": "적정성 판정",
      "tokens": ["부족", "충분", "넉넉", "적정", "모자라", "빈틈"],
      "context_tokens": ["공백", "비어있", "없어서"],   // 판정맥락에서만 차단(§2.3)
      "severity": "block"
    },
    "B2_switching": {
      "label": "갈아타기 권유",
      "tokens": ["갈아타", "리모델링", "바꾸세요", "해지하고"],
      "context_tokens": ["재설계"],
      "severity": "block"
    },
    "B3_safety_badge": {
      "label": "안전·심의 사칭",
      "tokens": ["안전", "안심", "심의완료", "보장확정", "검증완료", "적합판정"],
      "severity": "block"
    },
    "B4_certainty": {
      "label": "확정·단정",
      "tokens": ["무조건", "100%", "반드시", "절대", "확실히"],
      "severity": "block"
    },
    "B5_recommend": {
      "label": "추천·우열",
      "tokens": ["추천", "강추", "최고", "베스트", "유리합니다"],
      "context_tokens": ["1위", "낫습니다"],
      "severity": "block"
    }
  }
}
```

### 2.3 중립어/판정어 양의성 처리 — `context_tokens`

일부 토큰은 맥락에 따라 사실일 수도, 판정일 수도 있다. 단순 금지는 **거짓양성(legit 차단)** 을 부른다 — foliio 커뮤니티 PII 필터 트랩(과잉매칭) 계승.

| 토큰 | 사실 맥락(허용) | 판정 맥락(차단) | 처리 |
|---|---|---|---|
| 공백 | "치아보험 보장 공백 기간"(약관 용어) | "보장에 공백이 많아요"(판정) | `context_tokens` → §5 런타임에서 인접어 검사 |
| 없어요 | "암보험은 아직 없어요"(0원=사실) | "꼭 필요한데 없어요"(판정) | 화이트리스트 고정문구(§3)로만 발화 허용 |
| 보장 | "보유 보장 3건"(카운트) | "보장이 안전합니다"(사칭) | `보장`+`안전/확정` 연접 시 차단 |

> **원칙**: 양의성 토큰은 **블랙리스트로 막기보다 화이트리스트 고정문구로만 출력**한다. 자유생성 카피가 양의성 토큰을 쓰면 차단, 승인된 고정문구는 통과. 이게 거짓양성을 구조적으로 없앤다.

## 3. 화이트리스트 — 고객 노출면 허용 문구

고객뷰는 **자유 생성 금지**. 아래 **고정문구 카탈로그**에서만 발화한다(`neutral` 디폴트). 변수는 `{ }` 슬롯으로만 주입.

### 3.1 허용 문구 카탈로그

| ID | 문구 템플릿 | 슬롯 | 노출면 | 비고 |
|---|---|---|---|---|
| W-PAY-01 | `납입률 {pct}% (총 {total}회 중 {paid}회 납입)` | pct,total,paid | 공유뷰A | 사실 |
| W-PAY-02 | `이번 달까지 낸 보험료 {paid_amt}원 · 남은 보험료 {due_amt}원` | paid_amt,due_amt | 공유뷰A | 사실 |
| W-PAY-03 | `만기 {expiry_date}` | expiry_date | 공유뷰A | 사실 |
| W-COV-01 | `보유 담보 {name} · 보장금액 {amount}원` | name,amount | 공유뷰A | 사실(이름+금액) |
| W-COV-02 | `보유 {has}건 · 미보유 {none}건` | has,none | 공유뷰A | **충분/부족 카운트 숨김**, 객관 카운트만 |
| W-COV-03 | `{category} 담보는 아직 없어요` | category | 공유뷰A | **none(0원)만** 발화(severity=none) |
| W-CTA-01 | `정확한 진단은 담당 설계사와 확인하세요` | — | 공유뷰A | 권유 아님·확인 안내 |
| W-DISC-01 | (면책 고정문구 — §4.1) | — | 全 customer_facing | 상시·접기불가 |

### 3.2 발화 규칙 (insights 카피 — 정직성 레드라인)

공유뷰 상단 InsightHero의 '한 줄'은 **현재 `none`(0원)만 발화**. `enough`/`short`(graded)는 §7 단일스위치 플립 전까지 **출력경로 물리 부재**.

```
IF baseline_source IS NULL  → force neutral           # BE 권위(§7)
IF severity == 'none'       → W-COV-03 발화 가능
IF severity IN (enough,short) AND graded_enabled==FALSE → 발화 차단(경로 부재)
ELSE                        → InsightHero 미발화(헤드라인 숨김, 사실 표만)
```

**(미결 — blocking 승격 요청)** insights[] 발화 우선순위·표기상한(1개 vs Top3)·반올림 단위가 미확정. 고객노출=정직성 레드라인이므로 owner에 **준법 리뷰 추가** 필요(현재 PM+디자인만). 확정 전까지 **W-COV-03 단일 발화로 동결**.

## 4. 면책 카피 표준 — 고정문구

### 4.1 면책 고정문구 (W-DISC-01)

모든 `customer_facing` 화면 하단에 **상시 노출·스크롤/접기 불가**.

```
이 자료는 담당 보험설계사가 작성한 1차 보조자료입니다.
인파(Inpa)는 보험을 중개·권유하지 않으며, 보장의 적정성을 판정하지 않습니다.
정확한 진단과 가입 상담은 담당 설계사에게 문의하세요.
```

### 4.2 면책 변형 (노출면별)

| 노출면 | 면책 ID | 추가 문구 |
|---|---|---|
| 공유뷰A | DISC-SHARE | 기본문 + `표시된 금액·납입정보는 설계사가 등록한 데이터 기준입니다` |
| 비교안내서 §97 | DISC-COMPARE | 기본문 + `본 비교는 최종 판단 자료가 아니며, 계약 변경의 책임은 설계사와 계약자에게 있습니다` |
| AI 카톡 메시지 | DISC-KAKAO | 압축형 `※ 담당 설계사 작성 보조자료. 인파는 중개·권유하지 않습니다` |

### 4.3 면책 배치 규칙

- 면책은 **컴포넌트 구조에 하드코딩** — props로 on/off 불가(`DisclaimerFooter`는 항상 렌더, hideable prop 타입 부재 = 컴파일 차단).
- §97·M6은 면책 **상단+하단 2회**. 공유뷰는 하단 1회 고정.

## 5. 자동 검수 룰 — 3겹 게이트

카피가 고객 화면에 닿기까지 **3개 관문**을 통과한다. 어느 하나라도 실패 = 차단.

```
┌─ 작성(가이드) ─┐   ┌─ 컴파일(타입) ─┐   ┌─ 런타임(서버) ─┐
│ 카피 카탈로그   │ → │ 타입유니온 제외 │ → │ 응답 직전 스캔  │ → 고객 화면
│ 에서만 선택     │   │ = 빌드 차단     │   │ = 발송 차단     │
└────────────────┘   └────────────────┘   └────────────────┘
   사람 게이트          CI 게이트            방어선(최후)
```

### 5.1 Gate 1 — CI grep (정적 스캔)

빌드 파이프라인에서 **소스·i18n·고정문구 카탈로그**를 블랙리스트로 스캔. foliio 골든회귀(grep 100건 PASS) 계승.

```
검사 대상:  inpa_fe/src/**/*.{tsx,ts}  +  src/assets/i18n/*.json  +  compliance/*.json
대상 한정:  surface == customer_facing 로 태깅된 컴포넌트/번들만(planner_internal 제외)
규칙:       blacklist.tokens.json 의 tokens 완전일치 → FAIL(exit 1)
            context_tokens → WARN(리뷰 큐), 런타임 게이트로 위임
결과:       0건 PASS 시에만 머지 가능(blocking)
```

> **거짓양성 관리**: `planner_internal` 컴포넌트의 판정어는 스캔 제외(노출면 태깅으로 분리). allow-list는 **노출면 단위**로 관리 — 토큰 단위 예외 추가 금지(foliio 트랩: 패턴 느슨화 대신 화이트리스트 확장).

### 5.2 Gate 2 — 타입유니온 컴파일 차단

판정어·사칭어를 **타입 시스템에서 물리 제거**. 주석이 아니라 **컴파일 에러**로 막는다.

```
원칙:
  - DisclaimerFooter / ConsentBadge 등 customer_facing 컴포넌트 props 의
    문자열 유니온에서 "안전"·"심의완료"·"보장확정" 멤버를 물리 제외.
  - InsightHero severity prop = 'none' 만(현재). 'enough'|'short' 는
    graded 플립 전까지 유니온에 부재 → 코드가 발화 자체 불가.
  - 고정문구 ID 유니온(W-*)만 카피 슬롯에 주입 가능 → 자유 문자열 차단.
구조:
  type CustomerCopyId = 'W-PAY-01'|'W-PAY-02'|...|'W-DISC-01'   // 카탈로그 = 타입
  // 임의 string 을 카피로 넘기면 컴파일 에러
```

### 5.3 Gate 3 — 런타임 응답 스캔 (최후 방어선)

API 직렬화 직전, `customer_facing` 응답 페이로드를 블랙리스트로 최종 스캔.

```
위치:   share_token 응답 / §97 출력 / M6 메시지 직렬화 직전 미들웨어
입력:   surface 태그 + 페이로드 텍스트 필드
로직:   token 완전일치 → 발송 차단 + Sentry capture + 안전 폴백(빈 인사이트)
        context_tokens → 인접어 검사(§2.3) 후 차단/통과
의의:   타입·grep 우회(동적생성·AI출력)까지 잡는 그물.
        M5 ai_guardrail(M6 카톡)·§97 자동경고와 같은 계층.
```

### 5.4 검수 매트릭스 — 누가 무엇을 막나

| 위협 | Gate1 grep | Gate2 타입 | Gate3 런타임 |
|---|---|---|---|
| 하드코딩 금지어(소스) | ✅ | — | (보조) |
| i18n 키에 금지어 | ✅ | — | (보조) |
| 개발자가 임의 카피 prop 주입 | — | ✅ | ✅ |
| AI(Claude) 생성물에 금지어 | — | — | ✅(주력) |
| graded 조기노출 | — | ✅(유니온부재) | ✅ |
| `context_tokens` 양의성 | ⚠ WARN | — | ✅(인접검사) |

> **AI 생성물은 Gate3가 주력**. 정적 게이트로 못 잡으니 런타임이 최후 그물. §6에서 상술.

## 6. AI 생성물 게이트 — Claude 출력 후처리

### 6.1 왜 별도 게이트인가

§97 비교안내서(Opus)·M6 카톡 메시지·인사이트 카피가 **AI 자유생성**이면 블랙리스트가 무력화될 수 있다. grep·타입은 정적이라 못 잡는다. → **모든 AI 고객노출물은 M5 `ai_guardrail` 후처리 필수**(foliio `content_filter` 개조 계승).

### 6.2 ai_guardrail 파이프라인

```
Claude 출력
   │
   ▼
[1] 블랙리스트 토큰 스캔(B1~B5)  ── hit → 차단·재생성 or 폴백
   │
   ▼
[2] PII 정규식(전화/주민/카드/이메일)  ── foliio content_filter 재사용
   │
   ▼
[3] 단정·수익보장·비교과장 패턴  ── "무조건/100%/수익보장" 탐지
   │
   ▼
[4] referral 목적 시 "제3자 동의 확인" 문구 강제삽입  ── 무작위발굴 충돌 차단
   │
   ▼
[5] 면책 자동삽입(노출면별 §4.2)  ── 누락 시 append
   │
   ▼
통과분만 직렬화 → 클립보드(자동발송 0)
```

### 6.3 AI 게이트 적용 매트릭스

| AI 출력물 | 모델 | 게이트 단계 | 차단 시 동작 |
|---|---|---|---|
| §97 비교안내서 | Opus | [1][2][3][5] + §97 6항목 체크 | **발행 하드블록**(BE 거부) |
| M6 카톡 메시지 | (추정 Haiku/Sonnet) | [1][2][3][4][5] | 재생성 1회 → 실패 시 고정템플릿 폴백 |
| 인사이트 한 줄 | (현재 비-AI, 고정문구) | — | graded 플립 시 [1][3] 추가 |
| OCR 추출 텍스트 | Haiku | [2] PII만 | 정규화는 admin_verified만(자동승격 금지) |

### 6.4 planner_internal → customer_facing 누수 차단

설계사 내부면은 판정어 허용. 단 **공유 경로로 새는 순간 customer_facing 룰 적용**.

```
누수 지점:        히트맵 → 공유링크 / 기준설정 메모 → AI 메시지 인용 / 내부 코멘트 → 카톡
방어:            공유/메시지 직렬화 계층에서 surface 를 customer_facing 으로 강제 재태깅
                 → Gate3 런타임 스캔 자동 적용
원칙:            "내부면에서 쓴 판정어가 고객 페이로드에 복사되면 차단"
```

## 7. graded 플립 게이트 — 준법 단일스위치

`neutral`(none만 회색) → `graded`(enough/short 3색) 전환은 **부당권유 게이트의 핵심**. 이 플립이 본 문서 전체의 단일 통제점이다.

### 7.1 플립 선결조건 (AND — 전부 충족해야 플립 허용)

```
[ ] Q1: std_baseline 출처·권위 확정 (금감원/보험연구원/자체+면책 중 택)
[ ] planner_baseline 모델·기준설정 UI 존재 (충족 판정 주체 = 설계사)
[ ] enough/short 발화 시 면책문안 확정·배선 (§4)
[ ] insights 발화규칙(우선순위·상한·반올림) 준법 리뷰 통과
[ ] graded 카피의 화이트리스트 고정문구 추가 + 3겹 게이트 회귀
```

### 7.2 플립 메커니즘

```
전환점:   BE 1곳(heatmap_status mode 플래그 / baseline_source 권위)
FE 변경:  없음(재배포 불필요) — severity prop 유니온에 enough/short 추가는
          타입 릴리스 1회 선행, 이후 데이터로만 제어
실패측:   baseline_source==null → 영구 neutral(코드가 강제, FE 재판정 금지)
승인자:   준법(CPO=CTO) 명시 승인 게이트 — 출처+면책 없이 플립 금지
```

> **(blocking)** 위 선결 5종 미충족 시 graded 플립 **금지**. 베타는 `neutral` 동결 유지. 우회(조기 3색 노출)는 정직성 레드라인 위반.

## 8. 베타 미오픈 차단 플래그 — 법무 미확정 기능

법무 게이트(G3) 미확정 기능은 **명시적 플래그로 노출 0** 유지. "미구현"이 아니라 "차단됨"을 코드로 선언.

| 기능 | 플래그 | 베타 상태 | 해제 조건 |

| §97 비교안내서(M4) | `feature.compare_97` | **OFF(하드블록)** | G3 6항목 정본 + 대표 승인 |

> 플래그 디폴트 = **가장 안전한 차단 상태**. 누락/오류 시 안전측 실패(fail-safe). 해제는 준법 승인 후 1곳 토글.

## 10. 요약 — 룰의 3대 불변식

1. **고객면은 사실만, 자유생성 금지.** 화이트리스트 고정문구 카탈로그(§3)에서만 발화. 임의 string은 타입(Gate2)이 막는다.
2. **판정·권유·사칭은 3겹으로 막는다.** 작성가이드 → grep(Gate1) → 타입(Gate2) → 런타임(Gate3). AI 생성물은 Gate3가 최후 그물.
3. **graded 플립과 §97·referral은 준법 단일스위치.** 선결조건 AND 충족 + 대표 승인 전까지 fail-safe 차단(§7·§8). 누락 시 가장 안전한 상태로 실패한다.

> **본 문서의 위상**: dev/09 §6 후속과제 → **정본 승격 완료**. 자동검수 3겹 게이트(grep+타입+런타임)와 화이트/블랙리스트 SSOT(`blacklist.tokens.json` + 고정문구 카탈로그)를 명세하여 blocking 게이트로 동결. 미결 9종(§9)은 상위 정본으로 승격 요청.
